Go区块链模拟仿真大作业开发流程

一、项目初始化

1. 创建 Go 项目，完成 `go mod init`，建立基础目录，例如 `cmd/`、`internal/blockchain/`、`internal/wallet/`、`internal/network/`、`internal/gui/`、`data/`、`tests/`。
2. 统一基础配置，包括数据库路径、节点端口、创世区块参数、默认挖矿难度、区块奖励、日志输出格式。
3. 先搭建命令行入口，至少预留 `createblockchain`、`printchain`、`createwallet`、`getbalance`、`send`、`startnode` 等命令占位。
4. 约定序列化方案、哈希算法、签名算法、地址编码方案，避免后期结构反复调整。
5. 补齐最小可运行框架，要求项目能编译、能启动、能输出基础帮助信息。

验收：项目目录清晰，命令行程序可运行，后续模块可在现有骨架上继续扩展。

二、实现基础区块与区块链

1. 定义 `Block` 结构，至少包含区块高度、时间戳、交易列表、前一区块哈希、当前区块哈希。
2. 定义 `Blockchain` 结构，负责维护链的存储、创世区块创建、区块追加、链遍历。
3. 实现区块哈希生成逻辑，初期可先基于区块头字段做 SHA-256 计算。
4. 实现创世区块创建流程，保证项目首次启动时能自动初始化链数据。
5. 接入现代 KV 存储（推荐 Pebble），将区块和链元数据持久化保存，同时维护最后区块哈希索引。
6. 实现区块链迭代器，用于从最新区块向前遍历并打印整个链结构。
7. 完成 `addblock` 或内部追加逻辑，支持把新数据写成新区块。

验收：能够创建创世区块、持续追加新区块、重启后从数据库恢复链状态、打印完整链结构。

三、实现最简交易模型

1. 定义 `Transaction`、`TXInput`、`TXOutput` 基础结构，先不引入签名，只保留来源地址和目标地址。
2. 实现普通交易创建函数，支持输入地址、输出地址、金额，生成一笔可放入区块的交易。
3. 实现 coinbase 原型交易，作为创世阶段和后续挖矿奖励的基础。
4. 在打包新区块时把交易列表写入区块，而不是只存普通字符串数据。
5. 实现交易打印和交易哈希计算，便于后续签名、Merkle 树和网络广播复用。

验收：区块中可保存交易对象，能够构造一笔最简转账交易并写入新区块。

四、实现钱包系统

1. 定义 `Wallet` 结构，封装 ECDSA 私钥、公钥、地址生成逻辑。
2. 实现公钥哈希、版本字节拼接、校验和计算、Base58 编码，生成区块链地址。
3. 定义 `Wallets` 管理器，用于统一管理多个地址。
4. 实现钱包持久化，将钱包集合保存到本地文件，可使用 `gob` 或 `json`，但整个项目必须保持一种方案。
5. 程序启动时自动加载钱包文件，不存在时自动初始化空钱包集合。
6. 提供 `createwallet`、`listaddresses` 等命令行操作。

验收：可以批量创建钱包地址、重启后仍能读取地址列表、地址生成规则固定不变。

五、把交易升级为真实 UTXO 模型

1. 将交易输入改为引用历史交易输出，字段至少包含被引用交易 ID、输出下标、签名、公钥。
2. 将交易输出改为锁定到收款人公钥哈希，而不是直接保存明文地址。
3. 实现交易 ID 计算规则，确保每笔交易可唯一定位。
4. 在区块链遍历基础上实现 UTXO 查找逻辑，能够找到某地址所有未花费输出。
5. 实现余额查询函数，通过汇总 UTXO 余额返回账户金额。
6. 实现找零逻辑，创建交易时自动拼装输入输出，做到输入总额大于等于输出总额。

验收：系统能按 UTXO 模型创建交易，能正确查询余额，能生成找零输出。

六、实现交易签名与验证

1. 为交易增加签名流程，签名对象应是裁剪后的交易副本，不能直接对原始交易整体签名。
2. 发送交易时，先根据引用输出找到上一笔交易，再将对应输出的公钥哈希写入待签名副本。
3. 使用发送者私钥对交易哈希签名，并把签名结果写回每个输入。
4. 验证交易时，依次校验引用输出是否存在、输入公钥是否匹配被锁定的公钥哈希、ECDSA 签名是否有效。
5. 把交易合法性校验接入新区块打包和区块验证流程，禁止非法交易进入区块。

验收：只有持有正确私钥的用户才能花费 UTXO，伪造交易和错误签名会被拒绝。

七、实现 UTXO 缓存集

1. 独立设计 `UTXOSet` 模块，使用现代 KV 存储（推荐 Pebble）或独立前缀键保存每笔交易的未花费输出集合。
2. 在区块写入后同步更新 UTXO 集：删除已花费输出，写入新产生输出。
3. 提供 `FindSpendableOutputs`、`FindUTXO`、`Reindex` 等接口。
4. 在 `getbalance`、`send` 等命令中切换为优先读取 UTXO 缓存，而不是每次全链遍历。
5. 增加重建索引逻辑，防止缓存损坏后无法恢复。

验收：余额和可花费输出查询明显快于全链遍历，UTXO 集与链数据保持一致。

八、实现 Merkle 树

1. 为区块增加 `MerkleRoot` 字段。
2. 提取区块内全部交易哈希，按二叉树方式两两组合计算父节点哈希。
3. 若叶子节点数量为奇数，复制最后一个节点补齐后再继续构建。
4. 将最终根哈希写入区块头，并参与区块整体哈希计算。
5. 在区块验证流程中增加 MerkleRoot 重算校验，防止区块内交易被篡改。

验收：每个区块都能生成正确的 MerkleRoot，篡改交易后区块校验失败。

九、实现工作量证明共识

1. 为区块增加 `Nonce`、`Difficulty` 字段。
2. 定义目标值计算规则，使用难度决定哈希前导零或目标阈值。
3. 实现 PoW 挖矿器，循环修改 Nonce，直到区块哈希满足难度要求。
4. 生成新区块时先执行挖矿，再写库。
5. 节点验证新区块时重新校验区块哈希、Nonce、难度和区块内交易。
6. 在链管理层增加最长链原则，为后续分叉处理和网络同步打基础。

验收：新区块必须经过挖矿才能加入链，非法修改 Nonce 或哈希的区块会被拒绝。

十、实现交易池与经济模型

1. 建立 `Mempool`，用于缓存待打包交易。
2. 发送交易后先进入交易池，而不是直接写入区块。
3. 矿工打包区块时从交易池选择合法交易，按顺序写入区块。
4. 为普通交易增加手续费规则，手续费等于输入总额减输出总额。
5. 实现 coinbase 交易正式版本，要求每个新区块第一笔交易固定为矿工奖励交易。
6. 出块时把区块奖励和手续费一起支付给矿工地址。
7. 对交易池加入去重、双花预检查和无效交易清理逻辑。

验收：交易先进入交易池，再被矿工打包；区块第一笔始终为 coinbase；矿工能收到奖励和手续费。

十一、实现网络模拟

1. 设计节点结构，至少区分节点地址、已知节点列表、区块链实例、交易池实例。
2. 实现节点启动流程，支持通过命令行参数指定端口、是否开启挖矿、矿工奖励地址。
3. 设计消息类型，例如版本同步、区块请求、区块响应、交易广播、区块广播、地址广播。
4. 新节点启动后主动连接已知节点，请求对方最新区块高度和区块数据。
5. 当本地链落后时，按区块哈希或高度逐步同步并切换到更长链。
6. 当收到新交易时，先校验，再放入本地交易池，并向其他节点继续广播。
7. 当收到新区块时，先做完整性与共识校验，再更新区块链和 UTXO 集。
8. 如需加分，可补充 Gossip 式随机广播，或补充轻节点只下载区块头的 SPV 思路说明。

验收：至少两个节点能够启动、互相发现、同步区块、广播交易和广播新区块。

十二、实现安全校验与攻击模拟

1. 增加区块校验总入口，统一校验前一区块哈希、MerkleRoot、PoW、交易合法性。
2. 增加双花检测，同一笔 UTXO 若已在交易池或新区块中被使用，则拒绝第二次花费。
3. 增加输入输出金额校验，防止构造负数金额、超额输出或伪造奖励。
4. 编写双花攻击模拟案例，展示系统如何拒绝第二笔冲突交易。
5. 如时间允许，可实现简化版锁定脚本和解锁脚本，用于展示比单纯签名更细的花费条件控制。

验收：能够演示非法区块、非法签名、双花交易被拒绝的过程，并说明拒绝原因。

十三、实现 GUI 演示层

1. 确定前后端交互边界，后端提供区块、交易、钱包、节点状态等接口。
2. 使用 Wails 作为 Go 后端与前端桥梁，前端采用 React 和 MUI 构建页面。
3. 先做 Dashboard，展示当前区块高度、最新区块哈希、节点数量、交易池数量。
4. 实现钱包页面，展示地址列表、余额、创建钱包、发起转账。
5. 实现区块浏览页面，展示区块列表、区块详情、区块内交易详情。
6. 如需提高展示效果，可增加链式图形化视图，将每个区块按前后关系可视化。
7. GUI 的所有按钮操作都必须调用真实后端逻辑，不允许只做静态展示。

验收：系统可通过图形界面演示核心流程，包括创建钱包、查看余额、发送交易、查看区块和节点状态。

十四、性能对比与优化

1. 设计对比实验，至少包含“全链遍历查询余额”和“UTXO 缓存查询余额”两种方案。
2. 固定测试数据规模，例如区块数、交易数、地址数，保证实验可重复。
3. 记录每种方法的执行时间、查询次数和资源消耗。
4. 对热点模块做针对性优化，例如减少重复反序列化、批量写入数据库、减少无效网络广播。
5. 将实验结果整理为表格和结论，为答辩中的性能分析部分提供数据支撑。

验收：有可复现的实验方法和明确的性能对比结果，而不是只写主观描述。

十五、实验报告与答辩材料整理

1. 按模块梳理系统架构，包括区块、区块链、交易、钱包、UTXO、Merkle 树、PoW、交易池、网络、GUI。
2. 为每个核心模块整理数据结构、关键流程图、核心算法说明和关键代码截图。
3. 对开发顺序进行复盘，说明为什么要按“基础链路先打通，再逐步引入真实区块链机制”的方式推进。
4. 整理测试记录，包括正常交易、非法交易、双花攻击、节点同步、性能对比。
5. 准备最终演示脚本，确保现场可以依次展示建链、建钱包、转账、出块、同步、浏览区块和攻击拦截。

验收：报告内容完整，演示路径稳定，能够清楚说明每个模块的设计目的和实现效果。

十六、实现 Script / OP 虚拟机扩展

1. 设计简化版脚本结构，至少能表达推入数据和基础 OP 指令。
2. 实现最小 P2PKH 指令集，例如 `OP_DUP`、`OP_HASH160`、`OP_EQUALVERIFY`、`OP_CHECKSIG`。
3. 让交易输出从“只保存公钥哈希”升级为“保存标准锁定脚本”，同时兼容现有公钥哈希查询逻辑。
4. 让交易输入在花费时写入标准解锁脚本，用于提供签名和公钥。
5. 将交易验证流程升级为脚本执行验证，而不只是直接写死签名检查。
6. 保留对旧交易格式的兼容能力，避免前面阶段产生的数据无法继续使用。
7. 增加一个显式脚本展示命令，方便答辩时演示标准锁定脚本结构。

验收：系统能演示标准 P2PKH 脚本锁定/解锁，新交易验证经过脚本执行引擎，旧交易数据仍可兼容验证。

十七、实现脚本可视化与 GUI 演示增强

1. 为 GUI 钱包视图增加标准锁定脚本展示。
2. 为 GUI 区块和交易详情增加 `scriptSig` 与 `scriptPubKey` 展示。
3. 为 GUI 交易视图增加交易版本和是否启用 Script VM 的可视化标识。
4. 调整 GUI 后端的数据访问方式，避免长期持有链数据库句柄造成文件锁冲突。
5. 同步更新阶段文档、README 和答辩材料。

验收：脚本系统不仅在 CLI 和底层代码中可见，也能在 GUI 中直接演示，且 GUI 构建与运行稳定。

十八、实现分叉处理与最长链切换

1. 允许系统保存侧链区块，而不是只保存当前主链区块。
2. 导入新区块时校验其父块是否已存在，防止无父块孤儿区块直接进入主链。
3. 对分叉块在其父链状态上完成合法性校验，而不是只拿当前主链状态硬套。
4. 当新分叉长度超过当前主链时，自动切换到更长链。
5. 切换后重建 UTXO 集，保证余额和账本状态一致。
6. 增加显式 CLI 演示入口，能向老师展示最长链切换过程。

验收：系统能保存分叉块、识别孤儿块、在更长分叉出现时自动切换到最长链，并可通过 CLI 稳定演示。

十九、实现链重组后的 Mempool 恢复策略

1. 在最长链切换时识别旧主链独有的普通交易。
2. 识别新主链中已经确认的普通交易，避免它们继续留在 Mempool。
3. 对旧主链掉下来的、且在新主链下仍然合法的交易，重新加入 Mempool。
4. 增加显式 CLI 演示入口，能够向老师展示重组后交易被恢复到待打包状态。

验收：链重组后已确认交易会从池中清理，仍然合法的旧主链交易会重新进入 Mempool，并可通过 CLI 稳定演示。

最终交付建议

1. 命令行版本必须先完整跑通，再开始做 GUI。
2. 每完成一个阶段就做一次可演示的阶段验收，不要等全部功能写完再联调。
3. 所有新增模块都要优先复用前一阶段的数据结构，避免后期大范围重构。
4. 答辩时优先展示真实运行链路，再补充原理分析和扩展功能。
