# Plan 37：GUI 节点焦点面板与快捷操作

## 一、这一阶段要解决什么问题

到 Plan 36 为止，网络页已经能展示：

1. 网络拓扑
2. 统一事件时间线
3. 节点启动、连接、链控制
4. 多种网络场景流程

但实际操作时还有一个明显问题：

1. 节点多起来后，地址需要来回复制
2. 节点状态列表信息很多，但操作入口分散
3. 用户想对某一个节点继续操作时，缺少稳定的“当前焦点”

这会让网络页更像“信息堆叠”，而不是“运维工具”。

Plan 37 的目标，就是把网络页进一步收敛成一个真正可操作的工作台：

1. 增加节点目录
2. 增加节点焦点面板
3. 支持一键把当前节点地址填入连接表单、Seed 表单、链控制表单
4. 用聚焦操作替代大段冗余节点状态列表

## 二、为什么这一阶段值得做

前面几个阶段已经解决了“有没有能力”的问题，这一阶段解决的是“好不好操作”的问题。

当系统进入多节点阶段后，最常见的低效动作就是：

1. 先看节点地址
2. 再手动复制到连接区
3. 再手动复制到链控制区
4. 再回到节点状态里确认是不是选对了

这样的界面虽然能用，但不够像真实工具。

所以本阶段不是为了增加新的底层区块链机制，而是为了提升 GUI 的操作闭环：

1. 先选中节点
2. 再查看节点详细状态
3. 然后一键把节点填入对应表单
4. 最后继续执行连接、初始化、发交易或挖矿

这会让整个网络页从“展示页”更接近“控制台”。

## 四、核心理论讲解

### 1. 为什么网络页需要“焦点节点”

很多工具类界面都会有“当前上下文”或“当前对象”的概念。

例如：

1. 文件管理器里有当前选中的文件
2. 数据库工具里有当前选中的表
3. 运维面板里有当前选中的实例

区块链多节点工作台也一样。  
如果没有“当前焦点节点”，很多动作就必须依赖手工输入地址，使用成本会持续上升。

### 2. 为什么不能只保留节点列表

节点列表适合回答“现在有哪些节点”，但不适合承载完整操作流程。

因为真正操作时，用户更关心的是：

1. 当前节点是谁
2. 它是否已初始化
3. 它当前 tip 是什么
4. 它最近发生了什么
5. 我下一步要用它做什么操作

这类问题更适合通过“焦点面板”解决，而不是继续把信息塞进长列表里。

### 3. 为什么要提供表单快捷填充

工具类界面的一个核心目标，是尽量减少重复输入。

在这个项目里，节点地址会反复用于：

1. 连接已有节点
2. 作为新节点的 Seed
3. 作为链控制目标节点

既然这些表单都依赖同一个地址，就不应该要求用户每次手动复制粘贴。  
所以本阶段增加“快捷填入”按钮，本质上是在做操作链路压缩。

### 4. 为什么要把节点目录和焦点面板拆成独立组件

因为网络页已经比较复杂，如果继续在一个文件里叠加：

1. 节点目录
2. 焦点详情
3. 快捷操作
4. 节点状态逻辑

那后续会越来越难维护。

所以本阶段继续坚持低耦合拆分：

1. `NodeDirectoryCard.tsx` 负责节点列表选择
2. `FocusedNodeCard.tsx` 负责单节点详情与快捷操作
3. `NetworkTopologyCard.tsx` 负责拓扑视图与焦点联动
4. `NetworkPage.tsx` 只负责组合和状态连接

## 六、实现步骤

### 第一步：确定焦点节点状态

先在 `NetworkPage.tsx` 内部增加焦点节点状态，并在节点列表变化时保持它有效。

### 第二步：实现节点目录

增加独立目录卡片，用于：

1. 罗列节点
2. 高亮当前焦点
3. 让用户快速切换焦点

### 第三步：实现焦点面板

把原先分散在节点大列表中的信息收拢到单节点详情中，并加入快捷操作按钮。

### 第四步：让拓扑卡片参与焦点切换

点击拓扑节点卡片即可切换焦点，减少在不同区域之间来回定位。

### 第五步：移除冗长节点状态列表

用“节点目录 + 焦点面板”替代旧的大型节点状态区，减少重复信息。

## 七、关键代码讲解

### 1. `focusedNodeAddress`

这是本阶段最关键的页面状态。  
它把“当前在操作哪一个节点”显式表达出来，后续所有快捷动作都围绕它展开。

### 2. 焦点回退逻辑

当节点被停止、移除或当前焦点不存在时，系统需要自动选择下一个可用节点。  
否则界面会落入“用户看到焦点，但节点其实已经不存在”的不一致状态。

### 3. 快捷填充按钮

焦点面板里的按钮并不直接替代原有表单，而是负责：

1. 向连接表单写入本地节点地址
2. 向 Seed 字段写入目标地址
3. 向链控制表单写入当前节点地址

这样做的好处是：

1. 保留原始表单完整能力
2. 同时减少重复输入
3. 不破坏现有后端接口和提交流程

### 4. 拓扑与目录双入口

本阶段没有强制用户只能在一种地方切换节点，而是保留两条入口：

1. 拓扑图中选
2. 节点目录中选

这可以适应不同使用习惯，同时不增加后端复杂度。

## 八、常见错误

### 错误 1：只做节点目录，不做焦点面板

这样仍然会回到“选中了节点，但操作信息还是散落”的问题。

### 错误 2：快捷按钮直接绕过原表单提交

这样会让操作流程分裂，也不利于继续扩展参数。  
更合理的方式是“快速填入，再由原表单提交”。

### 错误 3：焦点节点失效后不自动回退

这样节点一旦被停止，界面会留下失效状态，用户体验会明显变差。

### 错误 4：重复保留旧大列表和新焦点面板

这样会让网络页再次变回重复堆叠，失去本阶段重构的意义。
