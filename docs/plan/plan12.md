# Plan 12：安全校验与攻击模拟

## 一、这一阶段要解决什么问题

前面的阶段已经让链具备了交易、签名、PoW、Mempool 和网络模拟，但还需要一层更明确的安全校验和攻击演示。

Plan 12 的目标是：

1. 更严格地校验交易合法性
2. 更严格地校验区块合法性
3. 明确拒绝双花
4. 提供可演示的攻击模拟入口

## 二、为什么安全校验要单独做成一个阶段

因为它不是某一个模块的小补丁，而是整个系统行为边界的收紧。

这一阶段要系统回答：

1. 什么交易必须被拒绝
2. 什么区块必须被拒绝
3. 双花如何被检测
4. 攻击案例如何稳定复现给老师看

## 三、本阶段范围

### 要做的内容

1. 增加统一交易安全校验入口
2. 增加统一区块安全校验入口
3. 拒绝非法签名交易
4. 拒绝非法引用输出
5. 拒绝双花交易
6. 拒绝非法区块导入
7. 增加 `simdouble`
8. 同步文档

### 这一阶段不做的内容

1. 不做脚本系统
2. 不做复杂拜占庭攻击模拟
3. 不做真实公网安全对抗
4. 不做 GUI 安全演示

## 四、核心理论

### 1. 双花是什么

双花就是同一个可花费输出被花了两次。

在当前系统里，双花至少可能出现在：

1. 同一个交易内部
2. 同一个区块内部
3. mempool 中两笔互相冲突的交易
4. 从网络导入的非法区块中

### 2. 为什么签名正确还不够

因为一笔交易即使签名正确，也可能：

1. 引用了已经花掉的输出
2. 在 mempool 中和另一笔交易冲突
3. 被打包到一个内部自相矛盾的区块里

所以必须把“签名合法”与“花费关系合法”分开校验。

### 3. 为什么要专门做攻击模拟

因为安全性不仅要写在代码里，还要能演示给老师看。

如果没有可复现的攻击演示，很多安全设计在答辩里就只能口头描述。

## 六、推荐设计

### 1. 交易校验

建议增加显式交易验证入口，至少检查：

1. 签名合法
2. 引用输出存在
3. 引用输出仍未花费
4. 同一交易内部没有重复花费同一输出

### 2. 区块校验

建议增加显式区块验证入口，至少检查：

1. MerkleRoot
2. PoW
3. coinbase 数量与位置
4. 区块内部无双花
5. 所有普通交易都合法

### 3. 攻击演示

当前阶段推荐优先做：

1. `simdouble`

让一笔交易先成功入池，再尝试构造第二笔冲突交易，并展示其被拒绝。

## 七、开发落地顺序

### 第一步：补交易安全校验

先把非法交易拦截边界补全。

### 第二步：补区块安全校验

让导入区块和本地出块都走统一安全规则。

### 第三步：补网络导入拒绝逻辑

让网络层对非法块和非法交易 fail closed。

### 第四步：增加 CLI 攻击演示入口

让双花攻击可以被显式演示。

## 八、CLI 如何演示

建议演示顺序：

1. 创建钱包
2. 创建链
3. `simdouble <from> <to1> <to2> <amount> [fee]`

展示重点：

1. 第一笔交易进入池中
2. 第二笔冲突交易被拒绝
3. 系统没有崩溃
4. 安全规则是自动生效的

## 九、验收标准

1. 非法签名交易被拒绝
2. 非法引用输出被拒绝
3. 双花交易被拒绝
4. 非法区块被拒绝
5. `simdouble` 可演示
6. 测试通过
7. 文档同步更新

## 十、常见错误

### 错误 1：只靠签名，不检查花费关系

这会让“合法签名的非法交易”漏过去。

### 错误 2：只在本地出块时检查，不在导入块时检查

这样网络层会成为绕过安全的入口。

### 错误 3：攻击演示只是打印提示，不真正触发系统拒绝

这样答辩时说服力不够。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段解决了正确性和运行性
2. 这一阶段开始把系统边界收紧，让非法行为明确失败
3. 双花和非法块不只是理论上不能过，而是代码里真的会被拒绝
