# Plan 22：最近链事件面板

## 一、这一阶段要解决什么问题

到 Plan 21 为止，系统已经可以展示“最近一次重组状态”，但这仍然只是一条单点信息。  
如果答辩里连续演示多次链切换、重组或恢复，老师还是会继续问：

1. 有没有最近几次事件的记录
2. 能不能看到事件顺序
3. 能不能在 GUI 里快速回顾这些链变化

Plan 22 的目标，就是把“最近一次重组状态”升级成“最近链事件面板”。

## 二、为什么这个阶段值得做

因为这一步的价值非常直接：

1. 把底层链变化做成更容易理解的历史序列
2. 让 GUI 不只是显示当前状态，还能显示最近发生过什么
3. 让 CLI 和 GUI 在讲解链切换行为时更有说服力

## 三、本阶段范围

### 要做的内容

1. 在区块链元数据中持久化最近链事件列表
2. 当前先记录重组 / 链切换事件
3. 增加 CLI 命令 `showevents [limit]`
4. 在 Dashboard 中展示最近链事件列表
5. 在网络页中展示最近链事件列表
6. 同步文档

### 这一阶段不做的内容

1. 不实现完整审计日志系统
2. 不做复杂图表或时间线组件
3. 不对每一个区块写入都产生日志事件
4. 不做分布式节点级事件聚合

## 四、核心理论讲解

### 1. 为什么“最近一次事件”还不够

最近一次重组状态只能回答：

1. 刚刚发生了什么

但不能回答：

1. 最近连续发生了哪些事件
2. 顺序是什么
3. 最近是否多次切换过主链

所以还需要一个“最近事件列表”。

### 2. 为什么采用固定长度事件列表

课程项目最重要的是：

1. 结构简单
2. 演示稳定
3. 易于解释

所以当前做法是保存一个固定长度的最近事件列表，而不是无限增长的日志表。  
这样更适合当前项目规模。

### 3. 为什么 Dashboard 和网络页都要显示事件列表

1. Dashboard 适合做全局总览
2. 网络页适合从节点和链行为角度解释事件

同一份真实后端数据，在不同页面承担不同讲解角色，这样最适合答辩。

## 六、实现步骤

### 第一步：扩展链元数据

增加最近链事件列表，并限制最大保留条数。

### 第二步：在重组发生时追加事件

当前先把 reorg 事件追加到最近事件列表中。

### 第三步：补 CLI 查询

增加 `showevents`，支持查看最近 N 条链事件。

### 第四步：补 GUI 面板

让 Dashboard 和网络页都能展示最近链事件列表。

## 七、CLI 如何演示

建议演示顺序：

1. `simreorg <from> <to> 20 1`
2. `showreorg`
3. `showevents 5`

演示重点：

1. 系统不仅记录了最近一次重组状态
2. 还保留了最近几次链事件的顺序记录
3. 这些记录来自真实链层元数据，而不是 GUI 假数据

## 八、验收标准

1. 最近链事件会被持久化
2. `showevents` 能正确输出最近事件列表
3. Dashboard 能显示最近链事件列表
4. 网络页能显示最近链事件列表
5. 测试通过
6. 文档同步更新

## 九、常见错误

### 错误 1：事件列表只存在内存中

这样刷新或重启后就丢失，不利于答辩演示。

### 错误 2：GUI 自己拼事件历史

正确做法应当是链层存储，GUI 只读真实数据。

### 错误 3：事件信息过少，无法解释行为

如果只有“发生过重组”而没有高度、tip、恢复数量，价值会大幅下降。

## 十、答辩时可以怎么讲

你可以这样解释：

1. Plan 21 解决了单次重组状态可观测性
2. Plan 22 进一步增加了最近链事件列表
3. 系统现在既能告诉你“刚才发生了什么”，也能告诉你“最近一段时间发生过什么”

## 十一、本阶段完成后的下一步

Plan 22 完成后，更适合继续推进的方向包括：

1. 更复杂的脚本类型
2. 更细粒度的链事件类型
3. 更接近 Bitcoin Core 的多区域信息组织
4. 更真实的 orphan / reorg 广播策略
