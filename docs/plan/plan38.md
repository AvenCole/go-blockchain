# Plan 38：GUI 控制台命令面板与预设工单

## 一、这一阶段要解决什么问题

到 Plan 37 为止，GUI 已经能：

1. 管理钱包
2. 管理节点
3. 运行网络流程
4. 在控制台里直接执行 CLI 命令

但控制台仍然存在一个典型问题：

1. 用户必须记住命令名
2. 用户必须自己拼接参数顺序
3. 多节点、多钱包场景下经常要来回复制地址
4. 虽然底层能力很完整，但入口仍然偏“记忆式”

这会让控制台更像一个原始输入框，而不是一个成熟工具。

Plan 38 的目标，就是把控制台升级成真正的“命令工作台”：

1. 增加常用命令预设分组
2. 根据当前钱包、节点、多签输出自动生成可执行命令
3. 支持一键填入命令
4. 支持一键直接执行预设命令
5. 支持快速复用最近执行过的命令

## 二、为什么这一阶段值得做

工具型 GUI 的目标，不只是“能调用后端”，还要尽量减少重复记忆和重复输入。

如果每次都要手工输入：

1. `createblockchain <wallet>`
2. `nodesend <node> <from> <to> 10 1`
3. `spendmultisig <signers> <txid> <out> <to> <amount> [fee]`

那控制台虽然强大，但使用门槛依然很高。

所以本阶段的关键价值，是把“命令行能力”转化为“工具式入口”：

1. 新手可以先点预设，再慢慢理解命令结构
2. 熟练用户仍然可以直接手输命令
3. 控制台既保留自由度，也提升了操作效率

## 三、本阶段范围

### 要做的内容

1. 为 GUI 控制台增加命令预设分组
2. 分组展示基础命令、钱包脚本命令、交易命令、节点命令、流程命令
3. 根据当前钱包、节点、多签输出来生成默认命令
4. 对参数齐全的命令允许一键执行
5. 对数据不足的命令允许先填入再手工修改
6. 展示最近执行过的命令，支持点击复用
7. 同步更新计划文档、README 和主流程文档

### 这一阶段不做的内容

1. 不做命令自动补全解析器
2. 不做 shell 级别多行脚本执行
3. 不做命令收藏持久化
4. 不做复杂参数表单替代整个 CLI

## 四、核心理论讲解

### 1. 为什么控制台需要“命令预设”

命令行系统的优点是灵活，但它的缺点也很明显：记忆成本高。

对于课程项目或桌面工具来说，更理想的做法是：

1. 底层保持命令行能力完整
2. 上层提供常用预设，降低使用门槛

这样既不会牺牲系统表达力，也能提升上手效率。

### 2. 为什么预设命令要基于当前数据生成

如果预设仍然写死为：

1. `<wallet-address>`
2. `<node-address>`
3. `<source-txid>`

那它只能算“文档提示”，还不算真正可执行的工具入口。

所以本阶段让预设命令读取当前：

1. 钱包列表
2. 节点列表
3. 多签输出列表

这样在大多数情况下，用户点一下就能直接执行。

### 3. 为什么要同时支持“填入”和“执行”

因为不同命令的使用场景不一样：

1. 有些命令已经具备完整上下文，适合直接执行
2. 有些命令只是接近可用，用户还想手动改参数

所以本阶段没有强行只保留一种入口，而是提供两种动作：

1. 填入输入框
2. 立即执行

这比单一入口更符合真实工具习惯。

### 4. 为什么最近命令复用也很重要

真实使用时，很多命令不是只执行一次，而是会反复微调。

例如：

1. `showevents 10`
2. `nodes`
3. `nodesend ...`
4. `nodemine ...`

如果最近命令能一键回填，用户就不需要反复重敲。  
这是一种很小但很实用的效率优化。

## 七、关键代码讲解

### 1. `buildConsolePresetGroups()`

这个函数负责根据当前上下文生成实际命令。  
例如：

1. 有钱包时生成 `createblockchain <wallet>`
2. 有两个节点时生成 `connectnode <node1> <node2>`
3. 有多签输出时生成 `spendmultisig ...`

这一步让预设从“说明文本”变成“可执行入口”。

### 2. `ready` 标记

每个预设都会带一个 `ready` 状态。  
它的作用是区分：

1. 当前可直接执行
2. 当前还缺少上下文，只适合先填入后手改

这能避免用户在数据不足时误触执行按钮。

### 3. `uniqueRecentCommands()`

这个函数负责对最近命令去重。  
如果不做去重，历史中重复运行的命令会把复用区刷满，反而影响效率。

### 4. 统一复用现有执行链路

本阶段没有新建一套“命令执行系统”，而是继续复用现有 `ExecuteCLI` 路径。  
这能保证：

1. 控制台手输命令
2. 预设命令直接执行
3. 历史命令复用

三种入口最终都走同一条后端链路，降低维护成本。

## 八、常见错误

### 错误 1：把所有预设都写死成占位符

这样看起来像有命令面板，但实际上并没有真正提升操作效率。

### 错误 2：预设直接绕开原命令执行通道

这样会形成两套行为路径，后续容易不一致。  
本阶段更合理的做法是继续走已有 CLI 执行接口。

### 错误 3：没有区分“可执行”和“需补参数”

这样用户会频繁点到不能跑的命令，体验会很差。

### 错误 4：最近命令不去重

这样复用区会被重复命令刷满，反而失去辅助价值。
