# Plan 24：更细粒度的链事件类型

## 一、这一阶段要解决什么问题

到 Plan 23 为止，系统已经有最近链事件面板，但事件类型还比较单一，主要集中在：

1. 重组 / 链切换

这意味着虽然能看到“大事件”，但老师如果继续问：

1. 正常出块有没有记录
2. 创世链建立有没有记录
3. 侧链块被保存时有没有记录

当前系统还不能非常完整地回答。

Plan 24 的目标，就是把最近链事件从“只有重组”升级为“覆盖主链增长、创世初始化、侧链保存等更细粒度事件类型”。

## 二、为什么这一阶段值得做

因为最近事件面板如果只有 reorg，很多时候会显得过于稀疏。  
而一旦补充更细粒度事件，系统就能更完整地解释：

1. 链是怎么建立起来的
2. 主链是怎么继续增长的
3. 侧链块是怎么被暂存的
4. 重组什么时候发生

这会让答辩中的“链行为讲解”更流畅。

## 三、本阶段范围

### 要做的内容

1. 记录创世初始化事件
2. 记录主链追加区块事件
3. 记录侧链块保存事件
4. 保留已有重组事件
5. 补充测试验证最近事件列表包含这些新类型
6. 同步文档

### 这一阶段不做的内容

1. 不对每个网络握手都做事件日志
2. 不记录无限制详细审计日志
3. 不增加复杂的事件过滤系统
4. 不做图表化时间线组件

## 四、核心理论讲解

### 1. 为什么要区分不同链事件类型

因为“重组”只是链行为的一部分。  
一个完整的教学型链行为观察，至少还应该看到：

1. 创世建立
2. 主链正常增长
3. 分叉块被存入侧链
4. 更长链切换造成的重组

### 2. 为什么主链增长事件很重要

如果最近事件里只有异常情况，而没有正常情况，老师会很难理解系统日常是怎么运转的。  
主链增长事件正好可以回答：

1. 平常出块是怎么记录的
2. 当前 tip 是如何推进的

### 3. 为什么侧链块保存事件也要记录

因为分叉处理本身就包含两个阶段：

1. 先把侧链块保存下来
2. 等侧链足够长时再切换

如果只记录切换，不记录侧链块存储，答辩时会缺一半故事。

## 六、实现步骤

### 第一步：在创世建立时追加 genesis 事件

让系统从一开始就有第一条链事件。

### 第二步：在主链追加区块时追加 main_block 事件

让正常链增长也成为可观察事件。

### 第三步：在侧链块被保存时追加 fork_block 事件

让最近事件列表能反映出分叉块的进入过程。

### 第四步：保留已有 reorg 事件

形成完整链行为序列。

## 七、验收标准

1. 创世建立后最近事件包含 `genesis`
2. 正常追加区块后最近事件包含 `main_block`
3. 侧链块保存后最近事件包含 `fork_block`
4. 重组后仍然保留 `reorg`
5. 测试通过
6. 文档同步更新

## 八、常见错误

### 错误 1：只有异常事件，没有正常事件

这样最近事件列表无法解释系统的日常行为。

### 错误 2：把主链块和侧链块混成一种事件

这样不利于答辩时解释“普通增长”和“分叉暂存”的区别。

### 错误 3：为追求细粒度而无限扩张事件种类

当前课程项目里，事件类型应该服务于讲解，而不是制造新的复杂度。

## 九、答辩时可以怎么讲
