# Plan 15：实验报告与答辩材料整理

## 一、这一阶段要解决什么问题

前 14 个阶段已经把系统做出来了，但课程大作业最后拿分不只看“代码能跑”，还看：

1. 能不能讲清楚
2. 能不能稳定演示
3. 能不能把实验结果组织成完整材料
4. 老师能不能快速理解系统结构与亮点

Plan 15 的目标，就是把已经完成的工程能力整理成一套真正可交付、可答辩、可复现的材料体系。

## 二、为什么这个阶段必须单独做

很多项目在开发完成后，最后分数不高，不是因为功能太少，而是因为：

1. 架构没有说明
2. 演示顺序混乱
3. 性能结果没有固定文件
4. 老师问到模块关系时答不清楚

所以 Plan 15 不是“补文档”，而是把工程成果转化为答辩成果。

## 三、本阶段范围

### 要做的内容

1. 增加系统架构说明文档
2. 增加实验报告文档
3. 增加答辩演示脚本文档
4. 增加答辩前检查清单
5. 对已有性能结果文件做引用整理
6. 把文档入口补到 README

### 这一阶段不做的内容

1. 不新增底层链功能
2. 不修改共识规则
3. 不重写 GUI
4. 不做新的网络协议扩展

## 四、核心思路

### 1. 把“代码结构”翻译成“老师能听懂的结构”

开发时你看到的是目录、函数、类型；答辩时老师更关心的是：

1. 模块之间怎么分工
2. 交易怎么走
3. 区块怎么生成
4. 安全如何保证
5. 为什么这样设计

所以必须把工程语言转换为讲解语言。

### 2. 把“结果”固定成文档和文件

性能实验、演示步骤、架构说明如果只在口头里，答辩时就很容易漏。  
Plan 15 的做法是把这些内容都固化成仓库中的正式文档。

### 3. 把“最终可演示链路”明确下来

答辩最怕临场忘顺序。  
因此需要提前定义：

1. 先演示钱包
2. 再演示建链
3. 再演示交易与挖矿
4. 再演示安全与网络
5. 最后演示 GUI 与性能结果

## 五、代码和文档会落在哪些位置

1. `docs/architecture.md`
2. `docs/report.md`
3. `docs/defense-script.md`
4. `docs/demo-checklist.md`
5. `docs/perf/latest.json`
6. `docs/perf/latest.md`
7. `README.md`

## 六、每份文档分别负责什么

### 1. `docs/architecture.md`

负责从系统级视角解释：

1. 模块边界
2. 数据主链路
3. 区块链核心模块组成
4. 钱包、网络、GUI 各自的位置

### 2. `docs/report.md`

负责形成实验报告主体，包含：

1. 实验目的
2. 总体设计
3. 数据结构
4. 核心功能
5. 性能实验
6. 不足与展望

### 3. `docs/defense-script.md`

负责定义一套稳定答辩脚本，让你知道每一步怎么讲、为什么讲。

### 4. `docs/demo-checklist.md`

负责在答辩前做最终排查，避免因为环境或数据问题导致翻车。

## 七、开发落地顺序

### 第一步：整理系统架构说明

先把模块关系写清楚，建立老师理解全局的入口。

### 第二步：整理实验报告主体

把前面阶段的实现内容转换为正式报告语言。

### 第三步：整理演示脚本

把 CLI、GUI、网络、安全、性能实验串成固定顺序。

### 第四步：整理检查清单

把环境、数据、命令、GUI、性能文件全部做成答辩前检查项。

## 八、验收标准

1. 仓库中存在完整架构说明
2. 仓库中存在实验报告主体
3. 仓库中存在答辩脚本
4. 仓库中存在答辩检查清单
5. README 能引导老师快速找到这些文档
6. 性能实验文件能直接引用

## 九、常见错误

### 错误 1：文档只是列目录，不解释设计原因

这样老师看完还是不知道你为什么这么做。

### 错误 2：演示步骤只写命令，不写讲解重点

这样你现场容易机械执行，讲不出重点。

### 错误 3：报告和代码脱节

如果文档和当前仓库状态不一致，答辩时很容易露出破绽。

## 十、答辩时可以怎么讲

你可以这样总结这一阶段：

1. 前 14 个阶段解决系统实现问题
2. Plan 15 解决交付与答辩问题
3. 目标是让系统不仅能运行，而且能稳定展示、清楚解释、方便复核

## 十一、本阶段完成后的下一步

Plan 15 完成后，最适合继续提升分数的方向就是：

1. 加入更像比特币的 Script / OP 虚拟机
2. 增强 CLI 与 GUI 的演示深度
3. 让“交易验证”从签名验证进一步升级为脚本执行验证
