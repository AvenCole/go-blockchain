# Plan 36：网络拓扑与演示时间线可视化

## 一、这一阶段要解决什么问题

到 Plan 35 为止，系统已经能完成：

1. 双节点同步演示
2. 双节点重组演示
3. 三节点分区 / 合流演示

这些功能在能力上已经很完整，但在答辩现场还有一个常见问题：

1. 老师看到按钮点完后，能知道结果
2. 但不一定能立刻看懂节点之间现在是什么连接关系
3. 也不一定能快速把“刚刚发生了哪些关键步骤”串成完整叙事

Plan 36 的目标，就是把当前网络页从“能跑 demo”提升到“能直接讲 demo”：

1. 用拓扑总览展示节点之间的连接关系
2. 用时间线把主链事件和节点事件合并到一个展示面板
3. 让答辩时的讲解顺序和界面信息顺序一致

## 二、为什么这一阶段值得做

因为高分答辩不只是展示功能是否存在，还要展示你是否把系统做成了“可解释的工程作品”。

如果界面上只有：

1. 高度
2. 是否重组
3. 一些零散日志

老师仍然要靠你口头补充很多上下文。  
而如果界面能直接告诉老师：

1. 当前有哪些节点
2. 它们互相怎么连接
3. 是否已经收敛到同一条链
4. 最近发生了哪些关键网络动作

那系统的表达力会明显提升。

## 三、本阶段范围

### 要做的内容

1. 在 GUI 网络页新增“网络拓扑总览”
2. 展示节点数量、连接数量、孤立节点数量、是否收敛
3. 展示每个节点的高度、tip、mempool、孤儿块数量、peer 列表
4. 展示节点之间的连接关系
5. 在 GUI 网络页新增“演示时间线”
6. 合并主链事件与节点事件为单一时间线
7. 用事件类型颜色帮助答辩时快速区分关键事件
8. 把实现拆到独立组件和工具函数中，降低网络页耦合
9. 同步本计划文档

### 这一阶段不做的内容

1. 不做真实图形化拖拽节点画布
2. 不做复杂动画连线
3. 不做历史事件持久化查询系统
4. 不做独立日志数据库

## 四、核心理论讲解

### 1. 为什么要做“拓扑视图”

因为区块链网络类项目，一旦进入多节点演示阶段，最容易让人混乱的就是：

1. 哪些节点在运行
2. 哪些节点互相连接
3. 哪些节点已经收敛

如果这些关系只能靠口头说明，老师会很难快速建立空间感。  
所以拓扑视图的意义不是“更好看”，而是“更容易理解网络结构”。

### 2. 为什么仅看高度不够

两个节点高度相同，并不说明它们就在同一条链上。

例如：

1. 节点 A 高度 2
2. 节点 B 高度 2

它们可能是：

1. 同一条链
2. 两条不同分叉

所以拓扑视图里必须继续展示 `tipHash`，并在总览区给出“是否收敛”的判断。  
这能让答辩时的结论更有证据。

### 3. 为什么要把主链事件和节点事件合并成一条时间线

因为网络演示本质上是一个过程，不只是一个结果。

例如一次三节点演示里，往往会先后发生：

1. 节点启动
2. 节点连接
3. 交易提交
4. 出块
5. tip 通告
6. 分叉
7. 重组
8. 收敛

如果这些信息散落在多个区域，老师会很难一下子抓住“刚刚系统到底做了什么”。  
所以时间线的价值在于把过程组织成一个可讲述、可回放的叙事结构。

### 4. 为什么这次要拆分前端组件和工具函数

因为网络页现在已经承载了很多职责：

1. demo 按钮
2. 链状态
3. 节点控制
4. 控制表单
5. 节点状态列表

如果再把拓扑和时间线逻辑继续堆在同一个文件里，就会明显变得难维护。  
所以本阶段把它拆到：

1. `NetworkTopologyCard.tsx`
2. `NetworkTimelineCard.tsx`
3. `networkView.ts`

这样页面层只负责组合，数据整理逻辑和展示逻辑分开，符合低耦合的最佳实践。

## 五、代码结构讲解

本阶段主要涉及：

1. `frontend/src/components/network/NetworkTopologyCard.tsx`
2. `frontend/src/components/network/NetworkTimelineCard.tsx`
3. `frontend/src/utils/networkView.ts`
4. `frontend/src/pages/NetworkPage.tsx`
5. `docs/plan/plan36.md`
6. `docs/plan/README.md`
7. `README.md`

### 1. `NetworkTopologyCard.tsx`

这个组件负责展示：

1. 总节点数
2. 总连接数
3. 孤立节点数
4. 是否收敛到同一条 tip
5. 每个节点的链状态和 peer 关系

它强调的是“网络空间结构”。

### 2. `NetworkTimelineCard.tsx`

这个组件负责展示：

1. 主链事件
2. 节点事件
3. 事件时间
4. 事件来源
5. 事件类别

它强调的是“网络时间顺序”。

### 3. `networkView.ts`

这是本阶段最重要的低耦合处理层。  
它负责：

1. 计算拓扑节点摘要
2. 去重并生成连接关系
3. 判断是否收敛
4. 生成统一的时间线数据
5. 提供地址 / hash 缩写工具

这样 UI 组件本身不需要承担复杂数据整理逻辑。

## 六、实现步骤

### 第一步：提炼网络展示工具函数

先把以下逻辑统一收敛到 `networkView.ts`：

1. 地址缩写
2. tip 缩写
3. peer 去重
4. link 生成
5. 收敛判断
6. 时间线排序

这样后续如果网络页还要继续增强，只需要复用工具函数，不必重新在组件里写一遍。

### 第二步：实现拓扑卡片组件

在拓扑组件中展示：

1. 总览状态
2. 节点卡片
3. 连接关系列表

这样老师可以同时看到：

1. 当前系统有几个节点
2. 节点分别是什么状态
3. 它们之间有没有互相认识

### 第三步：实现时间线组件

把：

1. `dashboard.recentEvents`
2. `nodes[].recentEvents`

统一合并，按时间倒序展示。  
这样就可以形成一个答辩时非常有用的“刚才发生了什么”列表。

### 第四步：接回网络页

在 `NetworkPage.tsx` 中只负责：

1. 传入节点数据
2. 传入主链事件
3. 组合拓扑卡片和时间线卡片

这样页面本身不需要处理复杂逻辑。

## 七、关键代码讲解

### 1. `buildTopology()`

这个函数负责从节点列表中整理出：

1. 节点摘要
2. 去重后的连接边
3. 是否收敛
4. 是否存在孤立节点

它是拓扑卡片的数据基础。

### 2. `buildTimeline()`

这个函数负责把主链事件和节点事件统一成一个数组，并按时间排序。  
这一步解决了“同一类演示信息分散在多个来源里”的问题。

### 3. `toneForKind()`

这个函数负责给不同事件分配视觉语义，例如：

1. `mine`、`tip_announce` 更偏成功状态
2. `reorg`、`orphan` 更偏警示状态
3. `connect`、`listen` 更偏信息状态

这样老师在看时间线时可以更快识别关键动作。

### 4. `NetworkPage.tsx` 的职责收缩

这一阶段后，网络页本身主要只负责：

1. 组合现有控制区
2. 组合新的可视化区
3. 传递 props

这意味着页面耦合度比继续把所有逻辑堆在一个文件里要低得多。

## 八、常见错误

### 错误 1：把拓扑数据整理逻辑直接写死在组件 JSX 里

这样组件会越来越难读、越来越难改。

### 错误 2：连接关系不去重

这样 A→B 和 B→A 可能会重复显示，反而让界面更乱。

### 错误 3：时间线只看主链事件，不看节点事件

这样会丢失网络层动作，演示仍然不完整。

### 错误 4：时间线只看节点事件，不看主链事件

这样又会丢失链级视角，老师还是很难把网络行为和链行为对应起来。
