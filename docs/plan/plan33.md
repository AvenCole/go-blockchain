# Plan 33：GUI 一键网络演示场景

## 一、这一阶段要解决什么问题

到 Plan 32 为止，GUI 已经能手动完成：

1. 启动节点
2. 初始化节点链
3. 通过节点发交易
4. 让节点挖矿

但如果答辩时间紧，老师通常希望看到一个“快速成型”的完整链路，而不是你手动点很多次按钮。  
Plan 33 的目标，就是把最典型的双节点同步演示做成“一键跑通”的快速场景。

## 二、为什么这一阶段值得做

因为课程答辩里，展示节奏很重要。

如果你需要逐步做很多准备动作，老师容易把注意力放在操作流程本身，而不是系统能力上。  
一键网络演示的价值在于：

1. 先快速证明系统真的能跑通
2. 再回到各个细节页慢慢讲原理

这会明显提升演示的完整度和观感。

## 三、本阶段范围

### 要做的内容

1. 在 GUI 中增加“一键网络演示”入口
2. 自动准备至少两个本地钱包
3. 自动启动两个 GUI 托管节点
4. 自动初始化主节点链
5. 自动让第二个节点连接主节点并同步创世块
6. 自动通过主节点发交易并挖矿
7. 自动等待 peer 同步到新区块
8. 返回演示结果摘要
9. 为 GUI 控制台增加 `runnetdemo` 命令
10. 同步文档

### 这一阶段不做的内容

1. 不做复杂多分叉随机网络实验
2. 不做图形化时间轴播放
3. 不引入新的共识算法
4. 不做节点自动清理策略

## 四、核心理论讲解

### 1. 为什么要把演示流程脚本化

因为系统功能越多，越需要一个稳定的“标准演示脚本”。  
一键场景本质上是把下面这条链路固化下来：

1. 节点启动
2. 节点同步
3. 交易广播
4. 区块产生
5. peer 高度更新
6. tip 通告出现

这能让系统能力更容易被重复验证。

### 2. 为什么这里选择双节点同步场景

双节点是最容易讲清楚、也最容易观察的最小网络单元。

它既能展示：

1. 主节点产生数据
2. peer 节点接收并同步

又不会像三节点或更多节点那样让演示信息过载。

### 3. 为什么需要等待条件判断而不是“发完就算成功”

因为网络同步和块广播是异步行为。  
如果只是发出命令后立刻返回，GUI 可能还没来得及刷新出最终结果。

所以这一阶段要明确等待：

1. peer 完成创世同步
2. peer 高度提升到新区块高度
3. 主节点产生 `tip_announce`

这样演示结果才更可信。

## 六、实现步骤

### 第一步：设计演示结果结构

需要把以下信息整理成结构化结果：

1. source 节点地址
2. peer 节点地址
3. 矿工地址
4. 接收方地址
5. 交易 ID
6. 区块 Hash
7. peer 同步到的高度
8. 是否观察到 tip 通告

这样 GUI 和控制台都能复用同一份结果。

### 第二步：实现服务层快速演示流程

在 GUI 服务层内部依次执行：

1. 补足钱包
2. 启动两个节点
3. 初始化主节点链
4. 连接 peer
5. 等待 peer 同步创世块
6. 发交易
7. 挖矿
8. 等待 peer 同步新区块
9. 检查 tip 通告事件

### 第三步：在 GUI 网络页增加一键演示卡片

用户点击按钮后，直接完成这条完整链路，并在页面里展示结果摘要。

### 第四步：为 GUI 控制台增加 `runnetdemo`

这样老师如果更喜欢看命令式演示，也能直接在 GUI 里的终端区域执行相同流程。

## 七、关键代码讲解

### 1. `RunNetworkQuickDemo()`

这是本阶段的核心编排函数。  
它不是新增底层协议，而是把已有的节点能力按固定顺序组合起来，形成一个稳定可重复的演示脚本。

### 2. `ensureWalletAddresses()`

这个函数确保演示至少有两个钱包可用。  
如果现有钱包数量不足，就自动创建并保存。

这样 GUI 演示不会因为“还没准备钱包”而中断。

### 3. `waitForNodeStatus()`

这个函数通过轮询节点状态来等待异步条件成立。  
它保证演示返回前，系统已经真正达到目标状态，而不是只把动作发出去。

### 4. `runnetdemo`

控制台命令复用了同一个服务层编排函数，因此：

1. GUI 按钮演示
2. 控制台命令演示

两条路径得到的是同一套行为，不会出现界面和终端逻辑不一致的问题。

## 八、验收标准

1. GUI 网络页有一键快速演示入口
2. 点击后可自动完成双节点同步、发交易、挖矿、peer 同步
3. 返回结果里能看到节点、交易、区块与同步高度摘要
4. GUI 控制台支持 `runnetdemo`
5. 测试通过
6. 文档同步更新

## 九、常见错误

### 错误 1：演示只完成节点启动，没有真正完成同步

这样不能证明网络链路真的跑通。

### 错误 2：演示返回太早，没有等待 peer 高度更新

这样 GUI 看起来像是成功了，但其实只是命令发出去了。

### 错误 3：GUI 按钮和控制台命令使用两套不同逻辑

这样后续维护很容易出现行为不一致。

### 错误 4：演示依赖人工提前创建钱包

这会降低稳定性，也不适合答辩现场快速重试。
