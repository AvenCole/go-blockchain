# Plan 35：三节点分区 / 合流重组演示

## 一、这一阶段要解决什么问题

到 Plan 34 为止，GUI 已经能一键演示：

1. 双节点正常同步
2. 双节点上的链重组
3. 已确认交易回滚并恢复到 Mempool

但老师继续追问时，往往还会问一个更进一步的问题：

1. 如果网络里不只两个节点呢
2. 如果有一部分节点先走旧主链，另一部分节点后来带着更长分叉回来呢
3. 最后所有节点能不能重新收敛到同一条最长链

Plan 35 的目标，就是把这个问题做成一个可直接答辩演示的三节点场景：

1. source 和 peer 先形成一个旧主链分区
2. fork 节点在隔离状态下保留并扩展另一条更长分叉
3. 三个节点重新合流后，最终全部收敛到同一个新 tip

## 二、为什么这一阶段值得做

因为双节点重组已经能证明“系统会切换最长链”，但三节点合流能进一步证明：

1. 这个能力不是只对单个 peer 有效
2. 节点之间会继续传播新的 tip
3. 最终网络可以重新一致

这在答辩里很加分，因为老师会更容易把它理解成“一个真正考虑了分区与恢复过程的教学型区块链网络模拟器”。

## 三、本阶段范围

### 要做的内容

1. 在 GUI 中新增“三节点分区 / 合流演示”
2. 保留 source + peer 旧主链
3. 克隆 source 的创世链数据，构造一个隔离 fork 节点
4. 让 source 先确认一笔交易
5. 让 fork 节点离线长出更长分叉
6. 重新连接两个分区，让 source 和 peer 都切换到 fork 分支
7. 检查三节点是否收敛到同一 tip
8. 在节点状态中显示 tip hash，便于观察收敛结果
9. 为 GUI 控制台增加 `runpartitiondemo`
10. 增加自动化测试
11. 同步文档

### 这一阶段不做的内容

1. 不实现真实网络断开 / 恢复按钮
2. 不做图形化消息时序图
3. 不实现 N 节点随机拓扑仿真
4. 不实现自动网络分区脚本语言

## 四、核心理论讲解

### 1. 为什么三节点比双节点更能说明“网络收敛”

双节点里，很多事情仍然像点对点同步。  
三节点之后，老师会更容易接受这是“一个小型网络”，因为你可以明确区分：

1. 旧主链分区
2. 隔离分叉分区
3. 合流后的全网一致

这说明系统关心的不只是单次同步，而是最终一致性。

### 2. 为什么这里用“复制创世链数据”来构造隔离分区

因为如果隔离节点完全独立初始化，它的创世块时间戳会不同，链就不是同一个起点。

而分区 / 合流演示要求的是：

1. 各节点最开始共享同一个祖先链
2. 只是后来在不同分区里各自向前发展

所以这里最合理的做法是：

1. 先创建 source 的创世链
2. 再把这份链数据复制给 fork 节点
3. 让 fork 节点从同一祖先块出发，离线长出更长分叉

这样得到的才是真正意义上的竞争分支，而不是完全无关的另一条链。

### 3. 为什么要显示每个节点的 `tipHash`

高度相同不代表真的收敛。

例如两个节点都在高度 2：

1. 可能是同一条链
2. 也可能是两条不同分叉刚好高度一样

所以要证明“三节点真的重新一致”，就必须看 tip hash。  
只有当三个节点的 `tipHash` 一样时，才能说明它们最终收敛到了同一条链。

### 4. 为什么还要保留 `restored` 结果

因为这个演示不仅要看网络收敛，还要继续证明：

1. 旧主链上那笔已经确认的交易
2. 在更长分叉替换后
3. 仍然能被正确恢复回 Mempool

也就是说，Plan 35 不是替代 Plan 34，而是在其基础上把网络规模再扩大一步。

这样 GUI 不再只能看高度，而是可以直接看节点当前链头是谁。

这一步非常关键，因为 Plan 35 的验收核心就是“是否收敛到同一个 tip”。

同时在每个节点卡片里显示 `tip`，便于答辩时直接观察最终一致性。

## 六、实现步骤

### 第一步：让节点状态返回 `tipHash`

先把底层节点快照扩展为：

1. 高度
2. Mempool 数量
3. 当前 tip hash

这样 GUI 和测试都能直接验证“是不是同一个链头”。

### 第二步：实现链数据目录复制

在 source 初始化完创世链后，把它的数据目录复制给 fork 节点。  
这样 fork 节点从一开始就和 source 拥有同一个祖先块。

### 第三步：实现三节点分区 / 合流编排

编排流程如下：

1. source 初始化链
2. peer 同步创世块
3. fork 节点从复制数据中启动，但先不加入网络
4. source 发送交易并挖矿，旧主链确认交易
5. fork 节点离线长出更长分叉
6. source 连接 fork 节点
7. source 因更长链发生重组
8. source 向 peer 通告新 tip
9. peer 请求缺失块并完成重组
10. 检查三节点 tip 是否一致

### 第四步：接入 GUI 和控制台

继续复用“服务层编排 + 界面调用 + 控制台命令”模式：

1. GUI 按钮适合课堂演示
2. 控制台命令适合老师要求你展示终端链路

这能保证展示方式不同，但底层行为一致。

## 七、关键代码讲解

### 1. `CopyDir()`

它把 source 节点的链数据原样复制给 fork 节点。  
从理论上说，这是为了保证：

1. 分叉发生在同一个祖先链上
2. 后续竞争块可以被正确识别为分支，而不是无关链

### 2. `RunNetworkPartitionDemo()`

这是本阶段的核心编排函数。  
它不仅负责启动三个节点，更重要的是固定了一个“可反复演示”的标准脚本。

答辩时你不需要临场手工构造网络分区，只要点一次按钮，就能把整个故事讲清楚。

### 3. `TipHash`

这是本阶段最关键的可视化指标。

因为老师看见：

1. 三个节点高度都到达目标高度
2. 三个节点 tip hash 一致

就能很直观地理解“网络重新收敛”这件事。

### 4. `runpartitiondemo`

这个命令让 GUI 内置终端也能直接跑同样的三节点脚本。  
这样如果答辩老师更偏好看命令式操作，你也不需要切回外部命令行。

## 八、常见错误

### 错误 1：用独立创世链代替共享祖先链

这样做出来不是“分区后的分叉”，而是两条互不相关的链。

### 错误 2：只看高度，不看 tip hash

这样无法证明最终真的收敛。

### 错误 3：只让 source 重组，不观察 peer 是否也同步切换

这样不能说明网络层已经把新的最长链继续传播出去。

### 错误 4：三节点演示和双节点演示使用完全不同的一套逻辑

这样后续维护会越来越混乱。  
所以本阶段继续复用已有 demo 编排能力，只在此基础上增加第三个节点和数据复制过程。

## 十、答辩时可以怎么讲

你可以这样解释：

1. Plan 34 证明了双节点上的最长链切换
2. Plan 35 进一步证明在三节点网络里，分区重新合流后也能重新收敛
3. 为了保证这是同一条链上的竞争分支，fork 节点不是独立创世，而是从 source 的创世链复制出来
4. 最终三个节点的 tip hash 一致，这说明网络重新达到一致状态
5. 同时旧主链交易仍能恢复到 Mempool，说明账本和交易池的一致性也被维护了
