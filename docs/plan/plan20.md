# Plan 20：GUI 网络运维台与内置命令控制台

## 一、这一阶段要解决什么问题

到 Plan 19 为止，底层系统已经具备：

1. 节点启动
2. 节点连接
3. 最长链切换
4. 链重组后的 Mempool 恢复

但 GUI 仍然主要只能展示：

1. Dashboard
2. 钱包
3. 区块
4. 交易与挖矿

这会导致一个问题：

1. 老师虽然能看到链和交易
2. 但不容易在 GUI 中直接演示“网络节点操作”和“命令行能力”

Plan 20 的目标，就是把 GUI 从“基础展示层”升级为“更接近 Bitcoin 客户端风格的运维与演示工作台”。

## 二、为什么这个阶段现在做最合适

因为当前项目已经有足够多的底层能力可以被可视化：

1. 节点状态
2. CLI 命令执行
3. 区块与交易状态
4. 链切换与同步行为

此时补网络页和控制台页，能明显提升答辩效果，而且不需要先去重做底层协议。

## 三、本阶段范围

### 要做的内容

1. 为 GUI 增加网络页面
2. 支持在 GUI 中启动本地节点
3. 支持在 GUI 中连接 seed 节点
4. 支持查看 GUI 托管节点的高度、矿工地址、peer 列表
5. 支持停止 GUI 托管节点
6. 为 GUI 增加控制台页面
7. 支持在 GUI 中直接执行 CLI 命令并查看 stdout/stderr/exitCode
8. 调整 GUI 主布局，让它更像客户端工作台
9. 同步文档

### 这一阶段不做的内容

1. 不实现完整交互式 shell 仿真
2. 不实现公网节点管理
3. 不做复杂的图表监控面板
4. 不做多窗口桌面架构

## 四、理论讲解

### 1. 为什么 GUI 里还要放控制台

课程答辩中，一个很常见的需求是：

1. 既想展示图形化界面
2. 又想证明底层 CLI 真能跑

如果 GUI 内置一个命令控制台，就能在同一个演示入口里同时展示：

1. 图形化页面
2. 真实命令输出
3. 后端调用链路

### 2. 为什么网络页很重要

链系统一旦做到网络模拟，仅靠命令行去解释节点状态，老师往往不容易一下子理解。  
网络页可以把这些信息直接展示出来：

1. 节点地址
2. 区块高度
3. 已知 peers
4. 是否绑定矿工地址

这会让“节点发现、连接、同步”这些概念更直观。

### 3. 为什么 GUI 托管节点要单独管理

GUI 的职责是演示和调度，而不是把所有网络逻辑塞进前端。  
因此更合理的结构是：

1. 前端页面只负责输入参数和展示状态
2. `internal/gui` 负责启动/停止/连接节点
3. 真实网络行为仍由 `internal/network` 执行

### 4. 为什么布局也要一起升级

如果新增页面还继续挤在原来的简单 tab 结构里，GUI 会越来越像“临时功能堆叠”。  
所以本阶段顺便把布局升级成更像客户端工作台的形式：

1. 左侧导航
2. 右侧内容区
3. 顶部状态摘要

## 六、实现步骤

### 第一步：补 GUI 后端接口

让 Wails 绑定能够暴露：

1. 执行 CLI
2. 启动节点
3. 停止节点
4. 获取节点状态
5. 连接 seed

### 第二步：补网络管理逻辑

在 `internal/gui` 中维护 GUI 托管节点会话，让页面可以查询和操作这些节点。

### 第三步：补控制台逻辑

通过 GUI 服务层调用现有 CLI，并把输出回传给前端。

### 第四步：补前端页面和布局

增加网络页与控制台页，并把主布局升级成更接近客户端工作台的结构。

## 七、验收标准

1. GUI 有网络页面
2. GUI 可启动节点
3. GUI 可连接 seed
4. GUI 可停止节点
5. GUI 可展示节点高度与 peers
6. GUI 有控制台页面
7. GUI 可执行 CLI 命令并显示输出
8. Go / 前端 / Wails 构建通过
9. 文档同步更新

## 八、常见错误

### 错误 1：控制台只是做个输入框，不真正调用 CLI

这样只是样子像控制台，不是真正的演示入口。

### 错误 2：前端直接控制网络层

正确做法应是通过 `internal/gui` 统一包装，再由它调用真实后端能力。

### 错误 3：新增页面后布局还是过于拥挤

这会让 GUI 继续显得像临时 demo，而不是完整工作台。

## 九、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段把底层链、网络和脚本能力做通了
2. Plan 20 把这些能力进一步做成一个更完整的客户端工作台
3. 老师现在既可以看页面，也可以直接在 GUI 内演示命令和节点操作
