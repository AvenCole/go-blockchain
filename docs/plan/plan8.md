# Plan 8：Merkle 树

## 一、这一阶段要解决什么问题

Plan 7 已经让交易、签名和 UTXO 缓存跑通，但区块还缺少一个专门用于描述“区块内交易集合完整性”的结构。

Plan 8 的目标是：

1. 为区块内全部交易构建 Merkle 树
2. 生成 `MerkleRoot`
3. 把 `MerkleRoot` 写入区块头
4. 提供基于重算的完整性校验能力

## 二、为什么 Merkle 树要单独做成一个阶段

因为它解决的问题和签名不同：

1. 签名证明的是“谁有权花费输出”
2. Merkle 树证明的是“区块内交易集合有没有被篡改”

所以它是区块完整性层面的增强，不应该和签名阶段混在一起。

## 三、本阶段范围

### 要做的内容

1. 提取区块内交易 ID 作为叶子
2. 构建 Merkle 树
3. 处理奇数节点补齐
4. 生成 `MerkleRoot`
5. 把 `MerkleRoot` 写入区块头
6. 更新区块哈希计算逻辑
7. 提供重算校验方法
8. 更新 CLI 打印信息
9. 同步文档

### 这一阶段不做的内容

1. 不做 PoW
2. 不做区块头完整共识验证
3. 不做 SPV 网络消息
4. 不做多节点 Merkle proof 传输

## 四、核心理论

### 1. Merkle 树是什么

Merkle 树是一种把多笔交易哈希逐层聚合成一个根哈希的结构。

### 2. 为什么区块里要有 MerkleRoot

因为区块头只需要保存一个根哈希，就能代表整组交易内容。

这意味着：

1. 修改任何一笔交易
2. 修改交易顺序
3. 删除或新增交易

都会导致根哈希变化。

### 3. 为什么奇数叶子要复制最后一个节点

为了保证每一层都能成对合并。  
这是区块链实现中常见的简化处理方式。

## 六、推荐设计

### 1. 叶子节点

直接使用交易 ID 作为叶子输入。

### 2. 父节点

把左右子节点拼接后再做哈希。

### 3. 区块头

在 `Block` 中增加：

1. `MerkleRoot`

并让区块哈希直接使用它，而不是临时交易拼接哈希。

## 七、开发落地顺序

### 第一步：实现 MerkleTree

先把树的构建过程独立出来。

### 第二步：让 Block 计算 MerkleRoot

让每个新区块在构造时就得到自己的根哈希。

### 第三步：更新区块哈希逻辑

让区块哈希直接引用 `MerkleRoot`。

### 第四步：实现完整性校验

提供重算根哈希并比对的方法。

### 第五步：更新 CLI

让 `printchain` 打印 `MerkleRoot`，便于演示。

## 八、CLI 如何演示

建议演示顺序：

1. 建链
2. 发交易
3. `printchain`

演示重点：

1. 区块头里已经包含 `MerkleRoot`
2. 每个区块的 MerkleRoot 都能展示
3. 篡改交易后理论上重算会失败

## 九、验收标准

1. 区块包含 `MerkleRoot`
2. `MerkleRoot` 随交易变化而变化
3. 奇数叶子情况可正常处理
4. `VerifyMerkleRoot()` 可工作
5. `printchain` 显示 `MerkleRoot`
6. 测试通过
7. 文档同步更新

## 十、常见错误

### 错误 1：直接继续用旧的交易拼接哈希

这样就没有把 Merkle 树作为独立结构落地。

### 错误 2：奇数节点不补齐

这会让树构建逻辑不稳定。

### 错误 3：有 MerkleRoot 但不做重算校验

这样等于只存了一个字段，没有真正建立完整性验证能力。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段解决了交易真实性和花费合法性
2. 这一阶段解决的是区块内交易集合完整性
3. 通过 MerkleRoot，区块头可以紧凑地代表整组交易内容

## 十二、本阶段完成后的下一步

到那时，系统已经具备：

1. 钱包
2. UTXO
3. 签名验证
4. UTXO 缓存
5. Merkle 树

这会让后续 PoW 和更真实的区块头校验更自然。
