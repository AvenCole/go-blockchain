# Plan 16：Script / OP 虚拟机

## 一、这一阶段要解决什么问题

到 Plan 15 为止，系统已经具备 UTXO、签名、PoW、Mempool、网络和 GUI，但交易花费条件仍然主要表现为：

1. 输出保存 `PubKeyHash`
2. 输入携带签名与公钥
3. 验证时直接用代码比较公钥哈希并校验 ECDSA 签名

这种做法能跑通教学链路，但还不够“像比特币”。

Plan 16 的目标，是把交易验证升级为“脚本驱动”的验证流程，让系统真正拥有一个教学型 Script / OP 虚拟机。

## 二、为什么这个阶段现在做最合适

因为前面的基础已经足够成熟：

1. 钱包和公钥系统已经稳定
2. 地址和公钥哈希规则已经固定
3. UTXO 引用关系已经稳定
4. 交易签名与验证已经跑通
5. CLI、GUI、网络层都已经可演示

这时引入 Script VM，不是在一片混乱中加复杂度，而是在已有稳定模型上把“花费条件表达方式”升级掉。

## 三、本阶段范围

### 要做的内容

1. 建立简化版 Script 数据结构
2. 建立脚本执行引擎
3. 支持最小 P2PKH 指令集
4. 让标准交易输出写入 `scriptPubKey`
5. 让标准交易输入写入 `scriptSig`
6. 用脚本执行替代新交易的直接签名验证路径
7. 保留旧交易格式兼容能力，避免历史数据失效
8. 增加 CLI 脚本展示入口
9. 同步文档

### 这一阶段不做的内容

1. 不实现完整 Bitcoin Script 全指令集
2. 不实现脚本循环、条件跳转等复杂语义
3. 不实现多重签名脚本
4. 不实现 Taproot、SegWit 等现代特性

## 四、核心理论讲解

### 1. 为什么比特币要用脚本，而不是只写死签名验证

如果系统只支持“签名正确即可花费”，那交易验证逻辑虽然简单，但可扩展性很差。

比特币把花费条件写成脚本，是为了让系统表达：

1. 需要哪个公钥
2. 需要满足什么哈希条件
3. 未来还可以扩展更多花费规则

所以脚本不是为了“更花哨”，而是为了把花费条件从硬编码逻辑提升为可表达的执行规则。

### 2. 什么是 P2PKH

P2PKH 就是最经典的“付款到公钥哈希”模式。它的锁定脚本可以理解为：

1. 复制输入提供的公钥
2. 对公钥做 HASH160
3. 与输出中保存的目标公钥哈希比较
4. 比较通过后，再检查签名是否正确

也就是说，输出不直接保存完整公钥，而是保存目标公钥哈希；输入在花费时再提供签名和公钥来解锁。

### 3. 本项目最小指令集为什么只选这几个

本阶段最少支持：

1. `OP_DUP`
2. `OP_HASH160`
3. `OP_EQUALVERIFY`
4. `OP_CHECKSIG`

这套组合已经足够覆盖标准 P2PKH 花费路径。  
对课程项目来说，这是最划算的一步：

1. 理论完整
2. 实现难度可控
3. 答辩时解释清楚
4. 又不会把系统复杂度拉爆

### 4. 为什么要保留旧交易兼容

因为项目前面阶段已经产生过旧格式交易。  
如果 Script VM 一上线就把旧数据全部搞废，实际使用和答辩演示都会出问题。

所以本阶段采用“新交易走 Script VM，旧交易保留兼容验证”的方式，让系统平滑升级。

## 六、关键实现思路

### 1. 输出结构仍保留 `PubKeyHash`

虽然系统已经加入 `scriptPubKey`，但这里没有把 `PubKeyHash` 直接删掉。原因是：

1. 余额查询仍然经常需要按公钥哈希判断归属
2. 这样可以减少对现有 UTXO 缓存逻辑的冲击
3. 兼容旧交易更容易

所以当前设计是：

1. 输出里保留 `PubKeyHash`
2. 同时增加 `scriptPubKey`
3. 两者共同服务于教学型实现

### 2. 新交易如何签名

新交易签名时，当前输入不会直接对“原始交易”签名，而是会：

1. 生成裁剪副本
2. 仅在当前输入位置临时放入被引用输出的锁定脚本
3. 对这个副本求哈希
4. 用私钥做 ECDSA 签名
5. 把签名和公钥组装进 `scriptSig`

这就更接近比特币签名语义了。

### 3. 新交易如何验证

验证时，系统会：

1. 取出输入的 `scriptSig`
2. 取出被引用输出的 `scriptPubKey`
3. 按顺序执行脚本命令
4. 在 `OP_CHECKSIG` 中验证签名与当前消息摘要
5. 栈顶最终为真时，交易输入才算合法

### 4. 为什么 CLI 要加 `showscript`

因为老师看到脚本结构时，如果只能听你口述，很难立刻建立直观印象。  
加入 `showscript <address>` 后，可以直接展示类似：

1. 目标地址
2. 公钥哈希
3. `OP_DUP OP_HASH160 ... OP_EQUALVERIFY OP_CHECKSIG`

这样答辩时会非常清楚。

## 七、开发落地顺序

### 第一步：新增脚本结构与最小 OP 集

先让系统具备表达脚本和执行脚本的基础能力。

### 第二步：让交易输入输出具备脚本字段

把标准交易切换到 `scriptSig + scriptPubKey` 模式。

### 第三步：把签名与验证切换到脚本驱动

这是本阶段最核心的一步。

### 第四步：补兼容与测试

保证旧交易仍能验证，新交易也能按脚本运行。

### 第五步：补 CLI 与文档

让功能不仅存在，而且能被稳定演示与讲解。

## 八、CLI 如何演示

建议演示顺序：

1. `createwallet`
2. `showscript <address>`
3. `createblockchain <miner-address>`
4. `send <from> <to> <amount> [fee]`
5. `printchain`

演示重点：

1. 地址能映射出标准 P2PKH 锁定脚本
2. 交易输入里能看到解锁脚本
3. 交易输出里能看到锁定脚本
4. 交易验证已经不只是“直接比对公钥哈希”，而是通过脚本执行完成

## 九、验收标准

1. 系统存在独立的脚本模块
2. 支持最小 P2PKH 指令集
3. 新交易输出可生成 `scriptPubKey`
4. 新交易输入可生成 `scriptSig`
5. 新交易验证经过脚本执行引擎
6. 历史旧交易仍可兼容验证
7. `showscript` 可在 CLI 中演示
8. 测试通过
9. 文档同步更新

## 十、常见错误

### 错误 1：把脚本做成字符串展示，而不真正执行

这样只是“看起来像脚本”，不是真的虚拟机。

### 错误 2：加入脚本后破坏旧数据

这会让系统升级成本过高，也影响答辩时的稳定性。

### 错误 3：把脚本和 UTXO 查询强耦合到一起

当前更合理的做法是保留 `PubKeyHash` 作为查询辅助，同时增加脚本字段，逐步升级，不要一次性推翻现有结构。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. 原来的系统已经能做签名验证，但验证逻辑还是代码直连式
2. Plan 16 把交易验证升级成了脚本执行模型
3. 当前实现的是教学型最小 Script VM，重点覆盖标准 P2PKH 路径
4. 这样系统更像比特币，也更容易解释“锁定脚本”和“解锁脚本”的关系

## 十二、本阶段完成后的下一步

Plan 16 完成后，如果继续往高分方向扩展，最合理的方向包括：

1. 更完整的最长链/分叉切换
2. 更复杂的脚本类型，例如多签
3. 在 GUI 中增加脚本可视化展示
4. 增加更完整的网络同步与状态面板
