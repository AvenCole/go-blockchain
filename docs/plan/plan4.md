# Plan 4：钱包系统

## 一、这一阶段要解决什么问题

Plan 3 已经让系统有了最简交易模型，但地址仍然只是普通字符串。  
Plan 4 的目标是把“字符串身份”升级成“真实加密身份”。

这一阶段要解决的核心问题是：

1. 如何生成钱包
2. 如何生成地址
3. 如何保存多个钱包
4. 如何在程序重启后恢复钱包

## 二、为什么钱包阶段必须现在做

后面的真实签名交易、UTXO、公钥哈希验证都依赖钱包系统。

如果现在不先把钱包做出来，后续阶段会缺少：

1. 私钥来源
2. 公钥来源
3. 地址生成规则
4. 身份持久化能力

所以 Plan 4 的作用是：先把“链上谁是谁”这件事建立起来。

## 四、核心理论

### 1. 钱包是什么

钱包本质上不是余额容器，而是一组密钥材料：

1. 私钥
2. 公钥
3. 基于公钥派生出的地址

### 2. 为什么地址不能直接等于公钥

直接暴露公钥虽然也能标识身份，但不利于展示和管理。  
所以一般会先对公钥做哈希，再做版本和校验，最后编码成更适合展示的地址字符串。

### 3. 为什么现在就要做持久化

如果钱包只存在内存里，一旦程序退出，身份就没了。  
后续做签名交易时就无法复用之前的钱包。

所以当前阶段必须先把钱包保存和加载这条链路打通。

## 六、推荐设计

### 1. `Wallet`

至少包含：

1. 私钥
2. 公钥

### 2. 地址生成流程

推荐流程：

1. 生成 ECDSA 密钥对
2. 对公钥做哈希
3. 加版本字节
4. 追加校验和
5. 做 Base58 编码

### 3. `Wallets`

使用一个管理器统一保存多个钱包地址到钱包对象的映射。

## 七、开发落地顺序

### 第一步：实现 Wallet

先打通密钥对生成。

### 第二步：实现地址生成

让钱包能输出可展示的地址字符串。

### 第三步：实现地址校验

确保不是任意字符串都能被当成合法地址。

### 第四步：实现 Wallets 管理器

支持多个钱包统一管理。

### 第五步：实现文件持久化

让程序重启后还能恢复之前创建的钱包。

### 第六步：接入 CLI

至少支持：

1. `createwallet`
2. `listaddresses`

## 十、常见错误

### 错误 1：把钱包当余额容器

钱包本质是身份系统，不是余额数据库。

### 错误 2：地址生成规则随意写

如果地址生成逻辑不稳定，后续签名交易和地址验证都会出问题。

### 错误 3：钱包不持久化

这样后面所有身份相关功能都会失去意义。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. 这一阶段把链上的身份从普通字符串升级为真实加密身份
2. 先建立钱包和地址系统，后面签名交易才能有真正的发送者与接收者
3. 当前先解决身份和持久化，签名验证放到后续阶段继续完成
