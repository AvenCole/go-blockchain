# Plan 40：网络流程进度反馈与运行锁

## 一、这一阶段解决了什么问题

到 Plan 39 为止，系统已经具备：

1. 网络拓扑总览
2. 统一事件时间线
3. 节点焦点面板
4. 链事件与节点事件筛选
5. 一键快速同步 / 重组 / 分区合流流程

但是网络页里仍然有一个很明显的工具化缺口：

1. 三类网络 demo 是长流程操作
2. 前端在执行过程中只能等待最终结果
3. 用户不知道当前流程运行到哪一步
4. 运行过程中仍可能重复点击冲突按钮
5. 节点启动、连接、初始化、发交易、挖矿也缺少明确的 busy 状态

Plan 40 的目标，就是把网络页从“按下按钮后等待结果”升级为“可观测、可锁定、可判断失败阶段”的工具工作台。

## 二、为什么这一阶段必须现在做

当前项目已经进入答辩可交付阶段。
这个时候再继续增加新功能，收益不一定大，但如果长流程操作仍然没有运行反馈，会直接影响：

1. 演示稳定性
2. 答辩时的可控性
3. GUI 作为桌面工具的完成度

所以 Plan 40 做的不是“继续堆能力”，而是把已有能力补到更稳定、可展示、可验证的状态。

## 三、本阶段实现范围

### 已完成内容

1. 为三类网络 demo 增加统一的运行进度事件模型
2. 后端在 demo 关键阶段发出实时进度事件
3. 前端订阅网络流程事件并展示流程状态卡片
4. 流程状态卡片展示：
   - 当前流程名
   - 当前阶段
   - 当前进度
   - 成功 / 失败状态
   - 错误信息
   - 开始时间 / 结束时间 / 耗时
5. demo 运行期间禁用其他 demo 按钮
6. 节点启动、连接、初始化、节点发交易、节点挖矿补充 loading / disabled 状态
7. 焦点节点中的停止节点按钮也接入运行锁
8. 增加自动化测试覆盖网络流程进度事件
9. 同步更新 README、任务总流程和计划目录说明

### 本阶段没有做的内容

1. 没有实现后台任务队列
2. 没有持久化任务历史
3. 没有增加新的链能力或新的网络 demo
4. 没有做跨页面全局任务中心

## 四、核心理论讲解

### 1. 为什么长流程必须有阶段反馈

对于桌面工具来说，“能执行”只是基础，“用户能不能知道系统正在做什么”同样重要。

如果一个操作需要几秒钟甚至更久，而界面只在最后弹出结果，那么用户会自然产生三个问题：

1. 是不是没点到
2. 是不是卡住了
3. 现在到底执行到哪一步

所以阶段反馈的本质是降低不确定性，而不是单纯做一个更花哨的提示条。

### 2. 为什么这次选择 Wails 运行时事件

当前三类 demo 的阶段都发生在 Go 后端：

1. 准备钱包
2. 启动节点
3. 初始化链
4. 节点连接
5. 提交交易
6. 挖矿
7. 等待同步 / 等待重组 / 等待收敛

这些真实阶段都在后端发生，所以最合理的做法不是让前端“猜测”流程，而是让后端把真实进度通过 Wails runtime event 推送给前端。

这样做的好处是：

1. 状态来源真实
2. 前后端职责清晰
3. 后续如果新增流程，也能复用同一套事件模型

### 3. 为什么还要补运行锁

只做进度提示还不够。
如果用户在流程执行中还能继续重复点击按钮，就可能造成：

1. 重复启动节点
2. 重复连接
3. 多个流程并发互相干扰
4. 演示结果不可预测

所以运行锁是进度反馈的配套机制。
它保证“当前正在执行的流程”拥有短时间的独占操作窗口。

### 4. 为什么节点操作只做 loading，而不做复杂状态机

像启动节点、连接节点、初始化节点链这些动作，相对来说是单步骤操作。
这类操作的重点不是细粒度阶段观测，而是：

1. 明确告诉用户它在执行
2. 避免重复点击
3. 执行完后恢复可操作状态

所以本阶段对节点操作采用局部 loading + disabled 即可，不必引入比 demo 更复杂的状态机。

### 1. `internal/gui/operations.go`

这是本阶段新增的后端进度跟踪层。
它负责：

1. 定义统一的事件名 `network:operation`
2. 封装 `networkOperationTracker`
3. 对流程执行中的开始、进度、完成、失败进行统一发事件

这样 `network_demo.go` 就不需要反复手写事件结构。

### 2. `internal/gui/network_demo.go`

这里是 Plan 40 的核心后端落点。
三类 demo 现在都会在关键步骤中显式发出进度事件，例如：

1. 钱包准备完成
2. 主节点启动完成
3. 对端节点启动完成
4. 节点连接完成
5. 原始确认区块同步完成
6. 更长分叉注入完成
7. 最终收敛完成

失败时也会带着当前阶段直接退出。

### 3. `internal/gui/service.go`

这里补了一层默认事件发射器包装。
它的目的有两个：

1. 在正常 Wails 生命周期里通过 runtime emit 发事件
2. 在测试环境没有有效 ctx 时自动降级为 no-op，避免测试报错

### 4. `internal/gui/types.go`

这里新增了 `NetworkOperationProgress`，作为前后端共享的进度结构：

1. `operation`
2. `status`
3. `phase`
4. `message`
5. `currentStep`
6. `totalSteps`
7. `startedAt`
8. `finishedAt`
9. `elapsedMs`
10. `error`
11. `summary`

### 5. `internal/gui/service_test.go`

这里新增了网络流程进度事件测试。
测试通过替换 runtime emitter 来捕获事件，验证：

1. demo 会发出 `started`
2. 最终会发出 `completed`
3. 最终事件的 `currentStep == totalSteps`

### 6. `frontend/src/App.tsx`

前端应用层新增了两类状态：

1. 当前网络流程状态 `networkOperation`
2. 一组本地 busy action 状态

同时还订阅了 Wails runtime event：

1. 接收后端发来的 `network:operation`
2. 更新网络页中的流程状态卡片
3. 在流程运行期间计算 demo 锁定状态

### 7. `frontend/src/components/network/NetworkOperationStatusCard.tsx`

这是本阶段新增的前端可视化组件。
它负责展示：

1. 当前流程名
2. 当前阶段
3. 进度条
4. 状态标签
5. 错误信息
6. 摘要
7. 耗时

它让“后端真实进度”变成“用户可以直接观察的工具界面”。

### 8. `frontend/src/pages/NetworkPage.tsx`

网络页接入了两类增强：

1. 三类 demo 按钮接入运行锁与运行中文案
2. 节点操作按钮接入 loading / disabled

这让网络页不再只是发起命令的界面，而是具备执行状态反馈的工作台。

## 六、实现步骤复盘

### 第一步：建立统一进度事件模型

先定义共享结构，保证前后端可以围绕同一份数据工作。

### 第二步：在后端 demo 中插入阶段事件

让原本只返回最终结果的长流程，补齐关键阶段的实时可观测性。

### 第三步：在前端订阅事件并展示状态卡

通过 Wails runtime event 把后端真实阶段显示到 GUI。

### 第四步：为节点操作补充本地 loading / lock

让短操作也具备稳定的交互反馈，避免重复点击。

### 第五步：补充自动化验证

通过 Go 测试、前端类型检查、前端构建、Go 构建和 Wails 构建，形成完整交付证据。

## 七、关键代码讲解

### 1. `networkOperationTracker`

它是本阶段最关键的抽象。
如果没有这个封装，每个 demo 函数都会充满重复事件发射代码，后续维护会很差。

### 2. 默认 emitter 包装

这里解决了一个很实际的问题：
Wails runtime 事件要求有效生命周期上下文，而测试环境没有。

所以这里采用：

1. 正常运行时：发真实事件
2. 测试环境：默认 no-op
3. 特定测试：允许注入自定义 emitter 捕获事件

### 3. 流程状态卡 + 本地 loading 的组合

Plan 40 没有强行把所有操作都统一成同一种复杂状态机，而是区分：

1. 长流程 demo：后端事件驱动
2. 单步骤节点操作：前端本地 loading

这种分层设计更符合当前代码复杂度，也更容易维护。

## 八、常见错误

### 错误 1：只在前端本地模拟 demo 进度

这样容易出现前端显示和后端真实阶段不一致的问题。

### 错误 2：只有成功提示，没有失败阶段

这样用户依然无法知道流程卡在哪一步。

### 错误 3：做了进度反馈，但没做运行锁

这样依然可能在运行过程中重复触发，导致状态混乱。

### 错误 4：测试环境直接调用 Wails runtime 事件

这会因为缺少有效 ctx 导致测试失败。
本阶段已经通过 emitter 包装解决这个问题。

## 十、答辩时可以怎么讲

这一阶段可以这样解释：

1. 前面几个阶段已经把多节点网络流程做出来了
2. 但长流程如果没有阶段反馈，演示时很难判断系统运行到哪一步
3. 所以这里补了真实后端进度推送和前端状态卡
4. 同时对关键按钮增加运行锁，防止重复触发
5. 这样网络页从“能点”提升到了“更稳定、更像工具”

## 十一、当前阶段结论

Plan 40 完成后，当前仓库已经具备一个比较完整、适合答辩展示的版本：

1. 底层链、交易、脚本、分叉、重组、多签能力完整
2. GUI 侧具备总控台、网络工具台、控制台和观测能力
3. 长流程操作已经有明确反馈和锁定机制

如果以“答辩基线”作为目标，当前阶段已经足够稳定。
