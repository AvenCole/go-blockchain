# Plan 40：网络流程进度反馈与运行锁

## 一、为什么下一步应该做这个

结合当前代码，最明显的下一步缺口不是底层链能力，而是**长流程操作的运行反馈**。

当前状态已经具备：

1. `internal/gui/network_demo.go` 中的三类网络流程：
   - `RunNetworkQuickDemo()`
   - `RunNetworkReorgDemo()`
   - `RunNetworkPartitionDemo()`
2. `frontend/src/pages/NetworkPage.tsx` 中的三组一键流程按钮
3. 事件时间线、链事件、节点事件筛选
4. 节点焦点面板、节点目录、控制台命令预设

但是当前实现仍然存在一个直接影响使用体验的问题：

1. 网络流程是串行长操作，前端只能等最终结果
2. `App.tsx` 里只在流程完成后设置 `message`
3. `NetworkPage.tsx` 的流程按钮当前没有运行中阶段反馈
4. 当前没有统一的“运行锁”，用户可在长流程执行中重复点击
5. 节点启动、连接、初始化、节点发交易、节点挖矿也缺少明确的 loading / busy 状态

这意味着系统虽然“能跑”，但还不够像一个稳定的桌面工具。

所以 Plan 40 最合理，目标也最明确：  
**把网络页从“点按钮等结果”，升级成“能看到阶段、能防止重复触发、能知道当前卡在哪一步”的工具工作台。**

## 二、本阶段目标

1. 给三类网络 demo 增加阶段进度反馈
2. 给关键节点操作增加运行中状态
3. 在操作执行期间对冲突按钮加锁
4. 让用户明确看到：
   - 当前正在执行什么
   - 执行到哪一步
   - 哪一步成功
   - 哪一步失败
   - 总耗时

## 三、为什么这比继续加新功能更优先

如果现在继续往下叠加更多演示功能，会出现一个问题：

1. 功能越来越多
2. 但长流程执行体验没有改善
3. 用户无法判断系统是在运行、等待还是卡住

这会直接影响：

1. 课堂展示稳定性
2. 答辩时的可控性
3. GUI 作为“工具”的完成度

也就是说，Plan 40 做的是**交互可靠性补强**，而不是表面功能堆叠。

## 四、代码依据

### 1. 后端长流程已经存在，但没有阶段输出

`internal/gui/network_demo.go` 里的三个 demo 方法本身已经天然分步骤，例如：

1. 准备钱包
2. 启动节点
3. 初始化链
4. 建立连接
5. 提交交易
6. 触发挖矿
7. 等待同步 / 等待重组 / 等待收敛

这些阶段在代码里已经是明确顺序，但目前没有对前端暴露过程状态。

### 2. 前端当前只有最终结果，没有过程状态

`frontend/src/App.tsx` 目前是：

1. 调用后端异步方法
2. 等待 promise 返回
3. 成功后显示 `message`
4. 然后 `refresh()`

这说明当前 GUI 只有“最终通知”，没有“过程观测”。

### 3. 网络页已经足够成熟，适合承接状态板

`frontend/src/pages/NetworkPage.tsx` 当前已经有：

1. demo 区
2. 拓扑区
3. 时间线区
4. 节点焦点区
5. 链事件区
6. 节点控制区

这说明网络页现在已经是最适合加“运行状态面板”的位置。

## 五、本阶段范围

### 要做的内容

1. 为网络 demo 建立统一的运行状态模型
2. 后端按阶段向前端发出进度事件
3. 前端新增网络流程状态卡片
4. 在状态卡片中展示：
   - 当前流程名
   - 当前阶段
   - 已完成阶段列表
   - 成功 / 失败状态
   - 错误信息
   - 开始时间 / 耗时
5. demo 运行期间禁用冲突按钮
6. 节点启动、连接、初始化、发交易、挖矿也补充 loading / disabled 状态
7. 失败时要能明确提示失败阶段
8. 成功后保留最终摘要
9. 同步更新文档

### 这一阶段不做的内容

1. 不做完整后台任务队列
2. 不做任务历史持久化
3. 不做多任务并发编排器
4. 不做跨页面全局通知中心

## 六、建议实现路径

### 第一步：定义统一进度事件模型

建议新增一套前后端共识的进度结构，例如：

1. `operation`
2. `phase`
3. `status`
4. `message`
5. `startedAt`
6. `finishedAt`
7. `elapsedMs`

前端和后端围绕这套结构同步。

### 第二步：在后端 demo 方法里发阶段事件

重点改动：

1. `internal/gui/network_demo.go`
2. 必要时新增 `internal/gui/operations.go` 或类似工具文件

做法：

1. 每完成一个关键阶段就发一次事件
2. 出错时发失败事件
3. 成功时发完成事件

### 第三步：前端订阅进度事件

重点改动：

1. `frontend/src/App.tsx`
2. `frontend/src/types.ts`
3. 必要时新增事件订阅工具层

让前端在 demo 执行过程中可以实时接收状态变化。

### 第四步：新增网络流程状态卡片

建议新增组件：

1. `frontend/src/components/network/NetworkOperationStatusCard.tsx`

放在网络页 demo 区附近，优先展示当前正在执行的流程。

### 第五步：加入运行锁

1. demo 运行时禁用其他 demo 按钮
2. 节点关键操作执行时禁用重复提交
3. 防止同一流程重复创建节点、重复触发连接或重复挖矿

### 第六步：补充验证

至少验证：

1. 按下流程按钮后会立即出现阶段反馈
2. 按钮在运行期间被禁用
3. 成功和失败路径都能正确显示
4. 流程结束后 UI 能恢复可操作状态

## 九、为什么这是 Ralph 视角下的最优下一步

Ralph 的核心不是“继续堆功能”，而是“朝着可验证完成的方向推进”。

从当前代码看，最值得补的是：

1. 运行反馈
2. 交互锁定
3. 操作可靠性

而不是再额外叠一个新 demo。

所以 Ralph 视角下，**Plan 40 = 网络流程进度反馈与运行锁**，这是最符合当前代码状态、最能提升可用性、也最适合下一轮直接落地的阶段。
