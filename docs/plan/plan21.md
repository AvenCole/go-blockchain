# Plan 21：链切换 / 重组状态可视化

## 一、这一阶段要解决什么问题

到 Plan 20 为止，系统已经能：

1. 处理更长分叉切换
2. 在链重组后恢复合法交易到 Mempool
3. 在 GUI 中展示钱包、区块、节点和控制台

但还有一个明显缺口：

1. 链切换和重组虽然已经真实发生
2. 但 GUI 中还没有一个明确的位置告诉老师“刚才发生了什么”

Plan 21 的目标，就是把链切换 / 重组状态从“底层逻辑”提升成“可视化、可讲解的观测信息”。

## 二、为什么这一阶段很重要

答辩里，老师通常不会只满足于“你说支持最长链切换”。

他们更容易继续问：

1. 最近有没有发生过重组
2. 从哪条链切到了哪条链
3. 切换时恢复了多少交易
4. 删掉了多少已确认池中交易

如果这些信息只能靠日志或口头解释，展示力还是不够。  
所以这一阶段的重点是让系统自己把这些状态讲出来。

## 三、本阶段范围

### 要做的内容

1. 在区块链层持久化最近一次重组状态
2. 记录旧高度、新高度、旧 tip、新 tip、恢复交易数、清理交易数、时间戳
3. 提供读取最近重组状态的接口
4. 增加 CLI 命令 `showreorg`
5. 让 GUI Dashboard 展示最近链切换状态
6. 让 GUI 网络页展示最近链切换状态
7. 同步文档

### 这一阶段不做的内容

1. 不实现完整历史事件日志系统
2. 不做可视化时间线图表
3. 不实现完整监控面板
4. 不做分布式节点级事件追踪

## 四、核心理论讲解

### 1. 为什么重组状态要持久化

如果最近一次重组只存在于内存里，那：

1. GUI 刷新后就可能丢失
2. CLI 无法独立查询
3. 答辩时不利于稳定演示

所以更合理的方式是把“最近一次重组状态”固化成链元数据的一部分。

### 2. 为什么只记录最近一次，而不是完整历史

课程项目里最重要的是：

1. 功能真实成立
2. 演示稳定
3. 结构清晰

完整事件历史虽然也有价值，但会明显增加设计复杂度。  
因此当前阶段只记录“最近一次重组”，是一个投入产出比很高的选择。

### 3. 为什么 Dashboard 和网络页都要显示

这两个页面的角色不同：

1. Dashboard 负责总览
2. 网络页负责解释链和节点行为

把重组状态同时放在这两个位置，老师会更容易在不同讲解路径中理解这个机制。

### 4. 为什么 CLI 也要加 `showreorg`

因为答辩时有时会先走 CLI，再切 GUI。  
如果 CLI 也能直接输出最近一次重组状态，就能证明：

1. 这不是 GUI 假数据
2. 这是链层真实记录下来的事件

## 六、实现步骤

### 第一步：在链层定义重组状态结构

先统一定义最近一次重组应该记录哪些字段。

### 第二步：在最长链切换时写入重组状态

切换发生时，把关键元数据写入数据库。

### 第三步：补 CLI 查询入口

增加 `showreorg`，让命令行可以直接查看最近一次重组信息。

### 第四步：补 GUI 展示

让 Dashboard 和网络页都能直观看到最近一次链切换状态。

## 七、CLI 如何演示

建议演示顺序：

1. `simreorg <from> <to> 20 1`
2. `showreorg`

演示重点：

1. 最近一次重组时间
2. 旧高度与新高度
3. 旧 tip 与新 tip
4. 恢复交易数与清理交易数

## 八、验收标准

1. 最近一次重组状态会被持久化
2. `showreorg` 可正确输出重组状态
3. Dashboard 可显示最近重组状态
4. 网络页可显示最近重组状态
5. 测试通过
6. 文档同步更新

## 九、常见错误

### 错误 1：只把信息打印到日志，不提供查询接口

这样不利于稳定演示。

### 错误 2：GUI 自己推导重组状态，而不是读真实后端数据

正确做法应当是链层持久化、GUI 只负责展示。

### 错误 3：只显示“发生过重组”，但不给具体细节

这样讲解时说服力不够。

## 十、答辩时可以怎么讲

你可以这样解释：

1. Plan 18 解决了链切换本身
2. Plan 19 解决了链切换后交易恢复
3. Plan 21 解决了链切换过程的可观测性
4. 现在系统不仅会重组，而且还能清楚告诉你最近一次重组发生了什么

Plan 21 完成后，更适合继续推进的方向包括：

1. 更复杂的脚本类型
