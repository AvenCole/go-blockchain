# Plan 28：GUI 脚本交易工作台

## 一、这一阶段要解决什么问题

到 Plan 27 为止，系统底层已经支持：

1. P2PKH
2. P2PK
3. 教学型多重签名

但这些脚本模板的主动创建和演示还主要依赖 CLI。  
如果 GUI 交易页仍然只能发送最普通的交易，那答辩时脚本能力的可视化展示仍然不完整。

Plan 28 的目标，就是把脚本模板选择能力直接带到 GUI 交易工作台中。

## 二、为什么这个阶段重要

因为当前 GUI 已经是答辩的主要演示入口之一。  
如果脚本能力只能在 CLI 中触发，那么：

1. 图形界面和底层能力之间会有割裂感
2. 老师会觉得 GUI 仍然只是普通交易展示层

所以这一阶段要让 GUI 也能主动构造：

1. P2PKH 交易
2. P2PK 交易
3. 教学型多重签名输出交易

## 三、本阶段范围

### 要做的内容

1. 为 GUI 后端增加脚本交易排队接口
2. 为 GUI 交易页增加脚本模板选择器
3. 支持 GUI 发起 P2PK 交易
4. 支持 GUI 发起教学型多签输出交易
5. 保持挖矿与 mempool 展示链路不变
6. 同步文档

### 这一阶段不做的内容

1. 不做 GUI 多签花费向导
2. 不做复杂脚本编辑器
3. 不做图形化脚本流程图
4. 不改底层 Script VM 规则

## 四、核心理论讲解

### 1. 为什么脚本模板也要在 GUI 里可选

如果 GUI 只有一种交易按钮，那它展示的只是“链能转账”。  
而当 GUI 能直接切换脚本模板时，老师就能更直观看到：

1. 同一条链支持不同锁定规则
2. 脚本系统不是隐藏在底层的死代码
3. GUI 正在调用真实后端脚本能力

### 2. 为什么先支持“多签输出创建”，而不是完整 GUI 多签花费

课程项目里更重要的是：

1. 先能清楚演示模板差异
2. 先让 GUI 能创建不同脚本输出

完整多签花费向导虽然也有价值，但交互复杂度更高。  
所以当前阶段先把最有展示价值的一步做进 GUI，更符合投入产出比。

## 六、实现步骤

### 第一步：补 GUI 后端接口

让 GUI 服务层能把脚本模板交易转换成真实后端调用。

### 第二步：补 Wails 绑定入口

把新的脚本交易能力暴露给前端。

### 第三步：补交易页模板切换表单

让交易页可以根据模板切换不同输入字段。

### 第四步：保留原有挖矿和 mempool 工作流

使 GUI 仍然沿用统一的“入池 -> 挖矿 -> 上链”演示链路。

## 七、验收标准

1. GUI 可切换交易脚本模板
2. GUI 可发起 P2PK 交易
3. GUI 可发起教学型多签输出交易
4. GUI 构建通过
5. 测试通过
6. 文档同步更新

## 八、常见错误

### 错误 1：GUI 只是切换标签，但后端仍然只走普通交易

这样不算真正支持脚本交易工作台。

### 错误 2：把脚本规则拼在前端里

正确做法应当是前端只收集参数，真实脚本交易仍由 Go 后端构造。

### 错误 3：为了做 GUI 脚本交易而破坏原来的普通交易流程

当前更合理的方式是扩展，而不是替换。

## 九、答辩时可以怎么讲

你可以这样解释：

1. Plan 27 把多签等脚本能力做到了链层和 CLI
2. Plan 28 进一步把脚本模板选择能力带到了 GUI
3. 现在 GUI 不只是展示交易，还能主动构造不同脚本输出类型

## 十、本阶段完成后的下一步

Plan 28 完成后，更适合继续推进的方向包括：

2. 更接近 Bitcoin Core 的多窗口信息组织

4. 更细粒度的脚本与网络联合演示面板
