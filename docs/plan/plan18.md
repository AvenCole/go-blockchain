# Plan 18：分叉处理与最长链切换

## 一、这一阶段要解决什么问题

到 Plan 17 为止，系统已经有：

1. 区块
2. UTXO
3. 签名与 Script VM
4. 网络广播
5. GUI 与答辩材料

但链管理层仍有一个明显短板：

1. 只能很好地处理“新区块直接接在当前 tip 后面”的情况
2. 对分叉链和更长侧链的处理还不够完整

Plan 18 的目标，就是把系统从“只会顺着当前主链往后长”升级为“能够保存侧链，并在出现更长分叉时切换到最长链”。

## 二、为什么这一阶段很重要

因为只要系统里存在网络同步，就一定会遇到一个经典问题：

1. 本地当前主链未必永远是最终主链
2. 其他节点可能发来另一条分叉分支
3. 当另一条分支更长时，系统应该切换到它

如果没有这层能力，网络模块虽然能广播区块，但链管理层仍然不完整。

## 三、本阶段范围

### 要做的内容

1. 允许数据库中同时保存主链块和侧链块
2. 导入新区块时校验其父块是否已存在
3. 支持导入不直接连接当前 tip 的分叉块
4. 当分叉链长度超过当前主链时，自动切换到更长链
5. 切换后重新索引 UTXO，保证账本状态一致
6. 增加 CLI 演示命令 `simfork`
7. 增加测试验证最长链切换行为
8. 同步文档

### 这一阶段不做的内容

1. 不实现完整的 orphan pool 持久缓存
2. 不实现复杂的链工作量比较规则
3. 不做真实比特币级别的重组广播策略
4. 不恢复旧主链交易到 mempool 的高级策略

## 四、核心理论讲解

### 1. 什么是分叉

分叉的本质是：

1. 两个不同区块引用了同一个父块
2. 于是从那个父块开始，链出现两条不同分支

在教学型区块链系统中，这通常出现在：

1. 节点延迟同步
2. 本地先形成一条短链
3. 后来收到另一条更长链的后续区块

### 2. 为什么需要最长链规则

如果系统不定义“哪条链才算主链”，那：

1. 当前余额无法确定
2. UTXO 集无法确定
3. 后续新区块不知道应接在哪条链后面

所以必须约定：

1. 当前选择更长链作为主链
2. 当更长链出现时，切换 tip
3. 切换后重新计算 UTXO 视图

### 3. 为什么要把侧链块也存下来

如果系统只保存当前主链块，那后来即使收到更长分叉，也无法从中间高度继续长出来。  
因此必须允许：

1. 侧链块先被存库
2. 暂时不成为主链 tip
3. 等后续侧链继续增长后，再触发切换

### 4. 为什么切换后要重建 UTXO

因为 UTXO 是“当前主链账本状态”的缓存。  
一旦主链 tip 发生切换，原主链上的花费关系可能就不再成立，所以必须：

1. 更新 last hash
2. 基于新主链重新构建 UTXO 集

这一步虽然比增量更新慢一些，但教学型系统中最稳妥、最容易讲清楚。

## 六、实现步骤

### 第一步：让链支持按任意 hash 回溯分支

先让系统能从任意区块 hash 向前回溯父块，拿到一条完整分支。

### 第二步：让导入逻辑接受侧链块

导入时不再要求新区块必须直接接在当前 tip 后面，只要父块已知即可进入校验流程。

### 第三步：在父链状态上验证新区块

新区块不应该只拿当前主链 UTXO 来校验，而是应该在其父分支的账本状态上校验。

### 第四步：更长链触发切换

当新分叉高度超过当前主链高度时，把 last hash 指向新 tip，并重建 UTXO。

### 第五步：增加 CLI 演示入口与测试

让这个能力不仅存在于底层代码，还能被稳定演示和验收。

## 七、CLI 如何演示

建议演示顺序：

1. `createwallet`
2. `createblockchain <miner-address>`
3. `addblock main-1`
4. `simfork <miner-address> 2`
5. `printchain`

演示重点：

1. 当前主链先有一条已有分支
2. 系统再构造一条更长的侧链
3. 当侧链更长后，tip 自动切换
4. 切换后打印链结构，展示当前主链已经改变

## 八、验收标准

1. 侧链块可以被保存
2. 未超过当前主链的分叉不会立刻切换
3. 更长链出现后会自动切换 tip
4. 切换后 UTXO 重建成功
5. `simfork` 可稳定演示
6. 测试通过
7. 文档同步更新

## 九、常见错误

### 错误 1：导入分叉块时仍强制要求必须接当前 tip

这样就不是真的分叉处理，只是普通追加。

### 错误 2：切换 tip 后不重建 UTXO

这样余额和账本状态会错乱。

### 错误 3：只在测试里支持分叉，不给演示入口

答辩时很难证明你真的做了最长链处理。

## 十、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段已经具备网络广播能力
2. Plan 18 开始补齐链选择规则
3. 系统现在不仅能接收新区块，还能在出现更长分叉时切换到最长链
4. 为了保证账本一致，切换后会重新索引 UTXO
