# Plan 30：Bitcoin Core 风格多区域工作台

## 一、这一阶段要解决什么问题

到 Plan 29 为止，GUI 已经功能很多：

1. 钱包
2. 区块浏览
3. 普通交易
4. 脚本模板交易
5. 多签创建与花费
6. 节点管理
7. 控制台

但这些能力如果仍然分散在多个页面里，答辩时老师需要来回切换，理解成本比较高。  
Plan 30 的目标，就是把主工作台做得更接近 Bitcoin Core / 桌面客户端的多区域信息组织方式。

## 二、为什么这一阶段值得做

因为课程答辩不只是展示“功能点有多少”，还要展示：

1. 你是否有一个清晰的主控台
2. 老师能不能一眼看到系统整体状态
3. 多个模块之间能否在一个页面里形成联动理解

所以这一阶段重点不是新增底层能力，而是把已有能力组织成更适合演示和理解的桌面工作台。

## 三、本阶段范围

### 要做的内容

1. 重构 Dashboard 为多区域工作台
2. 在一个页面中同时展示链总览、最新区块、钱包总览、Mempool、多签输出、节点活动摘要
3. 保持现有独立页面不丢失
4. 同步文档

### 这一阶段不做的内容

1. 不重写全部 GUI 页面
2. 不实现真正多窗口原生桌面布局
3. 不做复杂拖拽停靠系统
4. 不改底层链逻辑

## 四、核心理论讲解

### 1. 为什么要做多区域工作台

当系统功能增长后，一个“单区域 + 来回切页”的 GUI 会越来越难讲。  
多区域工作台的价值在于：

1. 当前链高度和最新区块可以同时看
2. 当前钱包和多签输出可以同时看
3. 当前 Mempool 和节点活动也能并排看

这样老师会更容易形成整体理解。

### 2. 为什么这更接近 Bitcoin Core 风格

Bitcoin Core 之类的桌面客户端给人的感觉通常是：

1. 信息密度高
2. 同一屏里能看到多个维度的状态
3. 它更像“系统控制台”，而不是单页 Web 应用

这正适合当前项目的答辩场景。

### 3. 为什么保留独立页面仍然重要

多区域工作台适合总览和串讲，独立页面适合深入细讲。  
两者不是替代关系，而是分工关系：

1. 工作台负责整体感知
2. 各功能页负责细节展开

## 五、代码结构讲解

本阶段主要涉及：

1. `frontend/src/pages/DashboardPage.tsx`
2. `frontend/src/App.tsx`
3. `docs/plan/plan30.md`

## 六、实现步骤

### 第一步：整理主工作台需要的状态来源

把链状态、钱包、Mempool、多签输出、节点活动摘要统一汇聚到 Dashboard 页面所需 props 中。

### 第二步：把 Dashboard 重构为多区域工作台

分成：

1. 顶部统计条
2. 最新区块 / 钱包总览
3. 链重组状态 / 最近链事件
4. 实时 Mempool / 未花费多签输出 / 节点活动摘要

### 第三步：保持其他独立页面继续可用

这样不会牺牲原有页面的可讲解性。

## 七、验收标准

1. Dashboard 成为多区域工作台
2. 一页中能同时看到链、钱包、Mempool、多签、节点活动摘要
3. 现有独立页面仍可正常使用
4. 构建通过
5. 文档同步更新

## 八、常见错误

### 错误 1：只是把原来内容堆得更满，没有结构

这样不会提高答辩可读性，只会更乱。

### 错误 2：为了工作台把所有独立页面删掉

这样会降低细节讲解能力。

### 错误 3：主工作台没有突出当前最重要信息

工作台的目的就是让老师快速看懂系统当前状态，而不是看一堆等权信息。

## 九、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段逐步把链、脚本、网络、GUI 能力做通
2. Plan 30 不再主要加底层功能，而是把已有能力重组成更适合答辩的主工作台
3. 老师现在可以在同一页同时看见多个关键维度的状态

## 十、本阶段完成后的下一步

Plan 30 完成后，更适合继续推进的方向包括：

1. 更真实的 orphan / reorg 广播策略
2. 更细粒度的脚本与网络联合演示面板
3. 更复杂的脚本策略组合
4. 更接近 Bitcoin Core 的高级信息组织方式
