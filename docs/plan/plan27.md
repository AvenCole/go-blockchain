# Plan 27：教学型多重签名

## 一、这一阶段要解决什么问题

到 Plan 26 为止，系统已经支持：

1. P2PKH
2. P2PK

这已经证明 Script VM 不只会跑一种模板，但还缺一个更有代表性的“多人共同授权”场景。  
Plan 27 的目标，就是在现有 Script VM 上加入教学型多重签名流程。

## 二、为什么这一阶段重要

多重签名是区块链课程中非常经典的脚本扩展案例。  
它能直接说明：

1. 资金不一定只由一个私钥控制
2. 花费条件可以要求多个参与者共同签字
3. Script VM 的价值不只是替代单签，而是表达更复杂的授权规则

## 三、本阶段范围

### 要做的内容

1. 增加教学型多签锁定脚本模板
2. 增加教学型多签解锁脚本模板
3. 支持创建多签输出
4. 支持多个本地签名者按顺序共同花费多签输出
5. 提供 CLI 演示命令
6. 补测试和文档

### 这一阶段不做的内容

1. 不实现完整 Bitcoin multisig 全语义细节
2. 不实现 P2SH 包装多签
3. 不做动态脚本编辑器
4. 不做任意复杂阈值策略组合

## 四、核心理论讲解

### 1. 多重签名的意义

多签的本质是：

1. 一笔输出不再只属于一个私钥
2. 花费它时，需要满足多个签名条件

这比普通单签更能体现脚本系统的表达力。

### 2. 为什么这里做“教学型多签”

课程项目里最重要的是：

1. 真实可跑
2. 容易讲清楚
3. 不引入过多协议细节

所以当前实现采用教学型多签：

1. 在锁定脚本中保存阈值和参与公钥
2. 在解锁脚本中按顺序提供签名
3. 由脚本引擎统一验证

### 3. 为什么顺序签名也有教学价值

真实比特币多签语义更复杂，但对于课程答辩来说，顺序签名已经足够说明：

1. 多个签名者共同控制输出
2. Script VM 需要处理多于一个签名
3. 交易花费条件已经明显超出普通单签

## 六、实现步骤

### 第一步：补多签脚本模板

让锁定脚本和解锁脚本可以表达“需要多个签名”。

### 第二步：补多签输出构造逻辑

让系统可以真正创建多签输出，而不只是打印示意脚本。

### 第三步：补多签花费逻辑

让多个本地签名者可以共同生成一笔有效花费交易。

### 第四步：补 CLI 演示入口

让答辩时可以直接走命令链路演示。

## 七、CLI 如何演示

建议演示顺序：

1. `createwallet` 多次生成参与者地址
2. `createblockchain <miner>`
3. `sendmultisig <from> 2 <addr1,addr2> 20 1`
4. `mine <miner>`
5. `spendmultisig <addr1,addr2> <source-txid> 0 <to> 10 1`
6. `mine <miner>`
7. `printchain`

演示重点：

1. 多签输出被成功创建
2. 花费该输出时需要多个参与者
3. 链中脚本结构已经明显不同于普通单签模板

## 八、验收标准

1. 系统支持教学型多签脚本
2. 系统能创建多签输出
3. 系统能花费多签输出
4. CLI 演示通过
5. 测试通过
6. 文档同步更新

## 九、常见错误

### 错误 1：只打印多签脚本，不真正验证多个签名

这样不算真正支持多签。

### 错误 2：把多签实现成独立交易系统

更合理的方式是扩展现有 Script VM，而不是另起一套逻辑。

### 错误 3：为了追求完整性把实现做得过重

当前阶段最重要的是教学型正确性和可演示性。

## 十、答辩时可以怎么讲

你可以这样解释：

1. Plan 26 证明了 Script VM 支持多模板
2. Plan 27 进一步证明它能表达多人共同授权的花费条件
3. 这让脚本系统从“单签扩展”升级为“更复杂授权策略”的雏形

3. 更细粒度的脚本与网络联合演示面板
