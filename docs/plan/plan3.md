# Plan 3：最简交易模型

## 一、这一阶段要解决什么问题

Plan 2 已经让项目具备了区块和区块链主干，但区块里存的还只是非常原始的内容。  
Plan 3 要解决的问题是：让区块开始真正承载“交易”。

这一阶段的目标不是直接做真实区块链中的签名交易系统，而是先做一个可运行、可演示、可讲解的最简交易模型。

## 二、为什么要先做“最简交易”而不是直接做完整交易

因为完整交易系统后面还会依赖：

1. 钱包
2. 地址编码
3. 公钥哈希
4. 签名与验签
5. UTXO

如果现在就把所有内容一次性混进来，复杂度会突然爆炸，前后阶段也会失去层次感。

所以 Plan 3 只做一个“教学上合理、结构上自然”的过渡层：

1. 先定义交易结构
2. 先让区块能存交易
3. 先让 CLI 能演示转账
4. 先让余额能以简化方式被计算

## 三、本阶段范围

### 要做的内容

1. 定义 `Transaction`
2. 定义 `TXInput`
3. 定义 `TXOutput`
4. 实现交易哈希
5. 实现 coinbase 原型交易
6. 实现普通无签名交易
7. 把区块从“存字符串”升级为“存交易列表”
8. 在 CLI 中支持最简转账与余额查询
9. 更新文档和演示说明

### 这一阶段不做的内容

1. 不做签名
2. 不做公钥系统
3. 不做钱包文件管理
4. 不做 UTXO 引用与查找
5. 不做手续费
6. 不做交易池

## 四、核心理论

### 1. 为什么交易要独立成结构体

如果区块里继续直接存字符串，那么后面钱包、UTXO、签名、Merkle 树都很难自然演进。

把交易独立成结构体后，后续所有阶段都能在这个结构上继续扩展。

### 2. 为什么现在先用“无签名模型”

这一阶段的重点是建立交易对象本身，而不是直接解决所有安全问题。

所以当前交易只表达：

1. 谁转出
2. 谁接收
3. 金额是多少

它是一个教学上的过渡版本，后面再升级成真实 UTXO + 签名模型。

### 3. 什么是 coinbase 原型交易

coinbase 原型交易可以理解为“系统生成的奖励交易”。

它的作用有两个：

1. 给创世区块一个天然的初始交易
2. 为后续经济模型和矿工奖励做铺垫

### 4. 为什么余额现在还能先做简化计算

虽然现在还没有 UTXO，但仍然可以先通过“输入减输出”的简化方式扫描链上交易，做出一个演示可用的余额结果。

这能帮助你更早把“交易 -> 区块 -> 链 -> 余额”这条主链路打通。

## 六、推荐的数据结构

### 1. `Transaction`

至少包含：

1. `ID`
2. `Inputs`
3. `Outputs`

### 2. `TXInput`

在当前阶段先简化为：

1. `From`
2. `Amount`

### 3. `TXOutput`

在当前阶段先简化为：

1. `To`
2. `Amount`

这种设计虽然不是真实 UTXO，但非常适合过渡到后续阶段。

## 七、开发落地顺序

### 第一步：先定义交易结构

把交易、输入、输出三个结构体先写出来。

### 第二步：实现交易哈希

让每笔交易都有稳定 ID，这样后面打印链和扩展 Merkle 树都会更自然。

### 第三步：实现 coinbase 原型交易

让创世区块和后续演示块都可以带有系统奖励型交易。

### 第四步：实现普通无签名转账

支持 `from -> to -> amount` 这条最简主流程。

### 第五步：让区块保存交易列表

把区块从单一数据字段升级为交易列表字段。

### 第六步：实现简化余额查询

按当前阶段的简化规则扫描交易，得到一个可演示的余额值。

### 第七步：接入 CLI

建议至少支持：

1. `send <from> <to> <amount>`
2. `getbalance <address>`
3. `printchain`

## 八、CLI 如何演示这一阶段

建议现场演示顺序：

1. `createblockchain miner`
2. `send miner alice 20`
3. `getbalance miner`
4. `getbalance alice`
5. `printchain`

这样老师能直接看到：

1. 链已经存在
2. 交易已经进入区块
3. 余额变化已经可见
4. 区块里打印出来的已经不是普通字符串，而是交易对象

## 九、当前阶段的简化边界

必须明确告诉老师，这一阶段是“最简交易模型”，所以它有意做了简化：

1. 地址还是普通字符串
2. 交易没有签名
3. 输入不是引用真实历史输出
4. 余额计算还是教学版扫描逻辑

这样不是偷懒，而是为了保证阶段递进清晰。

## 十、验收标准

Plan 3 完成后，至少要满足：

1. 区块里能保存交易对象
2. coinbase 原型交易可创建
3. 普通无签名交易可创建
4. `send` 能把交易写进新区块
5. `getbalance` 能返回简化余额
6. `printchain` 能打印交易明细
7. 测试通过
8. 文档同步更新

## 十一、常见错误

### 错误 1：现在就强行做真实 UTXO

这样会把 Plan 3 和 Plan 5、Plan 6 搅在一起。

### 错误 2：交易没有独立 ID

这样后面做交易打印、Merkle 树和签名升级都会很别扭。

### 错误 3：区块结构不升级

如果区块里还只是普通字符串，那后续几乎所有阶段都要返工。

### 错误 4：CLI 只做发送，不做余额和打印

这样老师看不到交易闭环，只能看到一个“像是执行了命令”的假效果。

## 十二、答辩时这一阶段可以怎么讲

你可以这样解释：

1. 这一阶段的目标是把区块中的内容从普通数据升级为交易对象
2. 但还没有直接进入真实加密身份和签名验证
3. 这样做的好处是能先把交易主链路跑通，再逐步升级安全性和真实性
