# Plan 31：版本通告与链同步提示

## 一、这一阶段要解决什么问题

到 Plan 30 为止，节点已经能够：

1. 广播区块
2. 处理孤儿块
3. 处理更长链切换

但还有一个偏真实网络行为的缺口：

1. 节点在接受到新区块或自己挖出新区块后
2. 应该主动向其他已知节点通告自己的最新高度

Plan 31 的目标，就是在已有教学型网络协议上补这一层“tip 通告”行为。

## 二、为什么这个阶段有价值

因为节点同步不只是：

1. 收到块
2. 写入链

还应该包括：

1. 让其他节点知道“我的最新高度变化了”

这样会让网络层更像一个真实会话系统，而不是单向消息收发。

## 三、本阶段范围

### 要做的内容

1. 节点在自己挖矿成功后，向其他 peer 发送最新 version 通告
2. 节点在导入新区块并推进 tip 后，向其他 peer 发送最新 version 通告
3. 增加最近网络事件里的 `tip_announce`
4. 增加测试验证 tip 通告事件会出现
5. 同步文档

### 这一阶段不做的内容

1. 不实现完整 inv/getdata 协议
2. 不实现复杂的广播退避策略
3. 不做链工作量比较广播
4. 不引入新的网络消息类型

## 四、核心理论讲解

### 1. 为什么节点要通告自己的最新高度

因为同步的关键不只是“请求数据”，还包括“告诉别人我已经变了”。

当一个节点：

1. 自己挖出新区块
2. 或导入新区块后 tip 前进

它有理由主动让其他节点知道自己的最新高度，从而触发后续同步逻辑。

### 2. 为什么这里继续复用 version 消息

当前项目的网络协议是教学型实现。  
为了保持简单和一致，本阶段不额外发明新消息，而是继续复用已有的 `version` 消息来传达：

1. 我是谁
2. 我当前的 best height 是多少

这足够支撑当前课程项目的演示需求。

### 3. 为什么这也适合加入最近网络事件

因为 tip 通告是一个很适合展示的网络动作。  
老师可以直观看到：

1. 节点接收了块
2. 节点更新了高度
3. 节点随后向 peers 发出高度通告

这让“同步提示”不再只是口头解释。

## 六、实现步骤

### 第一步：增加 tip 通告函数

让节点可以向除自身和指定来源以外的 peer 发送当前 version / height。  
这里直接复用已经拿到的最新高度，而不是为了发送通告再重新打开一次数据库。

### 第二步：在挖矿成功后触发通告

节点作为新区块生产者，应主动广播自己的最新高度。  
代码里可以直接使用刚挖出来区块的 `block.Height`，这样更直接，也能避免数据库句柄重复打开。

### 第三步：在导入并推进 tip 后触发通告

节点从别人那里接收块并更新 tip 后，也应主动通知其他 peer。  
这里要先比较导入前后的最佳高度，只有真的推进 tip 才广播。

### 第四步：记录最近网络事件

把 tip 通告事件写进最近网络事件流，方便 GUI 展示和答辩讲解。

## 七、关键代码讲解

### 1. `MinePending()`

这个函数在本地挖矿成功后已经拿到了新区块对象，因此最适合直接：

1. 广播新区块
2. 再把新区块高度作为 tip 通告高度发给其他 peer

这样不会重复读取数据库，也更符合“我刚挖出来，所以我知道自己高度是多少”的逻辑。

### 2. `importOrBufferBlock()`

这个函数负责处理接收到的区块。  
它先比较导入前的高度，再执行导入，最后再看导入后的高度是否前进。

只有 `afterHeight > beforeHeight` 时，才说明本地 tip 真的更新了，这时才应该触发 tip 通告。

### 3. `announceTipHeightExcept()`

这个函数负责把：

1. 当前节点地址
2. 当前最新高度

封装进 `version` 消息，然后广播给除自己和指定来源外的 peer。  
同时它会记录 `tip_announce` 事件，方便 GUI 网络页展示。

## 八、验收标准

1. 节点挖矿成功后会触发 tip 通告
2. 节点导入新区块推进 tip 后会触发 tip 通告
3. 最近网络事件里可看到 `tip_announce`
4. 测试通过
5. 文档同步更新

## 九、常见错误

### 错误 1：只更新本地高度，不做后续通告

这样网络同步会显得被动，不够真实。

### 错误 2：所有导入都盲目广播，不区分是否真的推进了 tip

应该只有在本地最佳高度前进时才做 tip 通告。

### 错误 3：tip 通告只做网络发送，不留事件记录

这样答辩时不容易证明这一步真的发生了。

### 错误 4：挖矿完成后为了发通告又重复打开数据库

如果底层 KV 数据库句柄还没有释放，特别是在 Windows + Pebble 这类场景下，重复打开数据库可能导致文件占用问题。  
所以更稳妥的做法是直接复用当前已经知道的最新高度。
