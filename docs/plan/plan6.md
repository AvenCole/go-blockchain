# Plan 6：交易签名与验证

## 一、这一阶段要解决什么问题

Plan 5 已经让交易变成 UTXO 风格，但当前系统还没有证明“谁有权花这些输出”。

Plan 6 要解决的核心问题是：

1. 发送交易的人必须用自己的私钥对交易签名
2. 节点在接收交易时必须验证签名是否有效
3. 输入中提供的公钥必须真的对应被花费输出的所有者

## 二、为什么签名验证要单独做成一个阶段

因为签名验证会同时依赖：

1. 钱包系统
2. UTXO 风格输入输出
3. 交易哈希
4. 引用历史交易输出

把这些内容单独做成一个阶段，能更清楚地说明：

1. 钱包负责身份
2. UTXO 负责花费对象
3. 签名负责授权证明

## 三、本阶段范围

2. 为 `TXInput` 增加公钥字段
3. 实现交易裁剪副本
4. 实现交易签名

6. 实现引用交易查找
7. 在入块前验证交易合法性

### 这一阶段不做的内容

1. 不做脚本系统
2. 不做手续费
3. 不做交易池
4. 不做网络广播签名同步

## 四、核心理论

### 1. 为什么不能直接对原始交易整体签名

因为交易本身包含签名字段。  
如果直接对带签名的完整交易签名，会形成循环依赖。

所以正确做法是：

1. 先构造一个去掉签名的裁剪副本
2. 把被引用输出的锁定信息填进去
3. 对这个副本求哈希
4. 再对哈希签名

### 2. 为什么输入里要放公钥

因为验证方需要知道：

1. 你宣称是谁
2. 你提供的公钥能不能推出正确的地址归属
3. 你的签名是不是由这把公钥对应的私钥产生的

### 3. 验证时到底检查什么

至少要检查三件事：

1. 输入引用的历史交易存在
2. 当前公钥对应的地址，确实匹配被花费输出的所有者
3. 签名对交易副本哈希是有效的

## 七、开发落地顺序

### 第一步：升级输入结构

先把签名和公钥字段加进去。

### 第二步：实现裁剪副本

让交易可以生成“去掉签名的待签名版本”。

### 第三步：实现签名

发送交易时，使用钱包私钥逐个输入完成签名。

### 第四步：实现验证

在区块写入前检查所有普通交易是否合法。

### 第五步：更新 CLI

让 `send` 真正使用钱包私钥签名，而不是只靠地址字符串拼交易。

## 九、验收标准

1. 普通交易可以签名
2. 普通交易可以验证
3. 篡改签名后验证失败
4. 篡改公钥后验证失败
5. 入块前会校验交易
6. 测试通过
7. 文档同步更新

## 十、常见错误

### 错误 1：对完整交易直接签名

这会让签名逻辑自引用，导致设计错误。

### 错误 2：只验签名，不验归属权

即使签名格式正确，也必须确认这把公钥确实对应被花费输出的所有者。

### 错误 3：CLI 发送逻辑不接钱包

如果 send 还只是用地址字符串，而不真正调用钱包私钥，那么签名阶段就没有真正落地。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. Plan 5 解决的是“花费哪一个输出”
2. Plan 6 解决的是“凭什么由你来花这个输出”
3. 钱包给出私钥和公钥，交易签名证明授权，验签逻辑证明交易没有被伪造
