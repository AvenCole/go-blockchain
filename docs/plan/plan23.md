# Plan 23：孤儿块缓冲与父块补取

## 一、这一阶段要解决什么问题

到 Plan 22 为止，系统已经具备：

1. 最长链切换
2. 链重组后的 Mempool 恢复
3. 最近链事件面板
4. GUI 网络运维台

但网络层仍然存在一个现实问题：

1. 节点收到新区块时，父块不一定已经先到达
2. 如果直接丢弃这个新区块，系统会丢失一次潜在同步机会

Plan 23 的目标，就是让节点支持“孤儿块缓冲”：

1. 暂时保存父块缺失的区块
2. 请求对方补发父链数据
3. 等父块到达后自动重试导入这些孤儿块

## 二、为什么这一阶段重要

真实区块链网络里，消息乱序是常态，不是例外。  
如果节点只能处理“父块总是先到、子块总是后到”的理想顺序，那网络模拟仍然过于乐观。

所以这一阶段是在补齐网络层的鲁棒性。

## 三、本阶段范围

### 要做的内容

1. 在节点中增加孤儿块缓冲结构
2. 当收到缺少父块的区块时先缓存，而不是直接失败
3. 向对端请求缺失父链数据
4. 当父块到达后自动重试其子孤儿块
5. 暴露当前孤儿块数量给 GUI 网络页
6. 增加网络层测试验证乱序到达场景
7. 同步文档

### 这一阶段不做的内容

1. 不实现完整 orphan transaction 池
2. 不实现无限增长的孤儿块数据库
3. 不实现复杂的网络带宽控制和重试退避
4. 不做真实比特币级别的 inv/getdata 流程

## 四、核心理论讲解

### 1. 什么是孤儿块

孤儿块就是：

1. 这个区块本身格式合法
2. 但它引用的父块当前节点还没有

这意味着它不能马上接入本地链，但也不应该立刻被视为无效块。

### 2. 为什么不能直接把孤儿块当成坏块

因为父块缺失只说明“本地视图还不完整”，并不说明这个块本身非法。  
如果一律丢弃，就会让节点对乱序消息过于脆弱。

### 3. 为什么孤儿块要与普通无效块区别处理

普通无效块应该拒绝，孤儿块应该暂存。  
这两类错误的语义不同：

1. 无效块：内容不可信
2. 孤儿块：上下文暂时不完整

### 4. 为什么 GUI 要显示孤儿块数量

答辩时，如果老师问“你怎么证明节点不会因为消息乱序直接失败”，  
GUI 网络页中的 orphan 计数就是非常直观的证据。

## 六、实现步骤

### 第一步：给节点增加孤儿块缓冲表

节点需要能够按：

1. 区块哈希
2. 父块哈希

追踪当前缓冲中的孤儿块。

### 第二步：在导入失败时识别孤儿块

如果底层导入返回“缺少父块”，则：

1. 先缓存这个块
2. 请求对方补发缺失父链

### 第三步：父块到达后重试子块

一旦父块导入成功，就检查是否有依赖它的孤儿块，并尝试递归补链。

### 第四步：暴露 orphan 计数到 GUI

网络页可直接显示每个 GUI 托管节点当前缓存了多少孤儿块。

## 七、验收标准

1. 子块先于父块到达时不会直接丢失
2. 父块到达后孤儿块会被自动重试导入
3. GUI 网络页能显示 orphan 数量
4. 网络层测试通过
5. 文档同步更新

## 八、常见错误

### 错误 1：把孤儿块和无效块混为一谈

这样会丢失本应能补回的区块。

### 错误 2：缓存了孤儿块，但父块到达后不重试

这样孤儿池就没有真正发挥作用。

### 错误 3：孤儿块状态只在底层，不暴露给演示层

这样答辩时很难证明这项能力真的存在。

## 九、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段解决了链切换和重组后的正确性
2. Plan 23 进一步提高了网络层对消息乱序的容忍度
3. 现在节点不会因为先收到子块就直接失败，而是会缓存孤儿块并等待父块到达
