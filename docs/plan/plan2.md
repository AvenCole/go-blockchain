# Plan 2：基础区块与区块链

## 一、这一阶段要解决什么问题

Plan 1 解决的是“项目骨架能不能跑起来”，Plan 2 开始解决“这个项目到底是不是一个真正的区块链程序”。

在这一阶段里，要先把最基本的区块链主干建立起来，让系统至少具备下面几种能力：

1. 能定义区块
2. 能定义区块链
3. 能生成创世区块
4. 能追加新区块
5. 能遍历整条链
6. 能把区块链状态持久化保存

这个阶段还不是完整区块链系统，但它会成为后面所有功能的主轴。

## 二、为什么 Plan 2 必须先打通基础链路

后面的交易、钱包、UTXO、Merkle 树、PoW、网络同步，全部都依赖一个前提：

区块链本身已经存在，而且结构稳定。

如果 Plan 2 没打牢，后面会出现这些问题：

1. 交易不知道挂在哪个区块上
2. 钱包无法查询链上余额
3. UTXO 无法从历史区块中扫描
4. PoW 无法确定作用对象
5. 网络同步没有统一的链结构可以传播

所以，Plan 2 的核心不是做“很多功能”，而是做“后续所有功能共同依赖的基础链路”。

## 三、本阶段范围

### 要做的内容

1. 定义 `Block` 结构
2. 定义 `Blockchain` 结构
3. 实现区块哈希计算
4. 实现创世区块创建
5. 实现新区块追加
6. 实现区块链遍历器
7. 实现基础存储
8. 建立命令行入口，至少能创建链、追加区块、打印链
9. 同步补充本阶段文档与代码讲解

### 这一阶段不做的内容

1. 不做真实签名交易
2. 不做钱包地址系统
3. 不做 UTXO 优化
4. 不做 Merkle 树
5. 不做 PoW
6. 不做网络广播

这样可以保证 Plan 2 只专注于“区块链主干”。

## 四、这一阶段的核心理论

### 1. 什么是区块

区块是区块链中的基本数据单元。一个区块通常至少包含：

1. 当前区块的数据
2. 当前区块的哈希
3. 前一个区块的哈希
4. 时间戳

把这些字段串起来，就形成了链式结构。

### 2. 什么是区块链

区块链本质上是一个按顺序连接的区块集合。  
每个区块通过 `PrevBlockHash` 指向前一个区块，这样就能保证顺序关系。

### 3. 为什么前哈希很重要

如果一个区块记录了前一个区块的哈希，那么一旦前面的区块被篡改，后面的链式关系就会被破坏。

这正是区块链“不可篡改性”概念的基础。

### 4. 为什么现在就要持久化

如果 Plan 2 只放在内存里，程序一关闭链就没了，后面的钱包、交易、UTXO、网络同步都没法往下接。

所以即使现在功能还比较少，也要先打通基础存储。

## 五、这一阶段代码会落在哪些位置

Plan 2 落地时，代码主要会进入这些位置：

1. `internal/blockchain/`
   - `block.go`
   - `blockchain.go`
   - `iterator.go`
   - 可能还会有序列化或存储相关文件
2. `internal/cli/`
   - 增加区块链相关命令入口
3. `cmd/go-blockchain/main.go`
   - 继续只做入口装配，不直接堆业务逻辑
4. `data/`
   - 存放链数据

## 六、推荐数据结构设计

### 1. `Block`

这一阶段推荐至少包含这些字段：

1. `Timestamp`
2. `Data`
3. `PrevBlockHash`
4. `Hash`
5. `Height`

这里的 `Data` 在 Plan 2 可以先简单处理，不急着换成完整交易结构。

### 2. `Blockchain`

推荐至少包含：

1. 最后区块哈希
2. 数据库存储句柄

这样后续可以快速找到链尾，并向后追加新区块。

### 3. 区块链迭代器

需要能从最新区块开始，逐个回溯前一个区块，直到创世区块。

这样做有两个目的：

1. 可以打印整条链
2. 后续可以支持交易扫描和 UTXO 遍历

## 七、开发落地顺序

### 第一步：先写 `Block`

先把区块结构体和区块哈希逻辑写出来。

这一阶段要先确保：

1. 能创建一个区块对象
2. 能根据字段生成哈希

### 第二步：写创世区块

创世区块是链的起点。

要单独提供创建函数，而不是把它和普通新区块创建混在一起。

### 第三步：写 `Blockchain`

把区块链结构建立起来，并提供基础初始化逻辑。

这一步的目标是：

1. 第一次启动时能创建创世区块
2. 再次启动时能从存储中恢复已有链

### 第四步：接入存储

把区块写入数据库，并维护“最后区块哈希”的索引。

这一步一旦打通，后面链的追加和恢复都会稳定很多。

### 第五步：写追加新区块逻辑

让系统支持把新数据打成新区块，挂到当前链尾后面。

### 第六步：写链遍历器

让系统能从链尾一路往前回溯，最终打印整条链。

### 第七步：接入 CLI

至少要补三个命令方向：

1. 创建链
2. 添加区块
3. 打印链

### 第八步：补测试和文档

这一阶段不应该只把代码写出来，还要同步补：

1. 结构说明
2. 哈希逻辑说明
3. 创世区块说明
4. 持久化说明
5. CLI 使用说明

## 八、命令行这一阶段建议做到什么程度

建议至少新增以下命令：

1. `createblockchain`

## 九、这一阶段的验收标准

Plan 2 完成后，至少要满足：

1. 能创建创世区块
2. 能追加新区块
3. 能从数据库恢复区块链状态
4. 能打印完整链结构
5. CLI 可以触发这些操作
6. 测试覆盖关键结构和主流程
7. 文档已经同步补齐

## 十、这一阶段的常见错误

### 错误 1：把交易系统提前混进来

Plan 2 还没有到真实交易阶段，不要在这一阶段就把签名交易、UTXO 等逻辑揉进来。

### 错误 2：创世区块和普通区块混为一谈

创世区块应该是单独的初始化逻辑，不能完全照搬普通区块创建方式。

### 错误 3：只做内存链，不做持久化

如果现在不打通基础存储，后面很多阶段都要返工。

### 错误 4：CLI 逻辑直接堆进 `main.go`

这样会破坏 Plan 1 刚建立好的结构边界。

### 错误 5：文档不解释哈希和链关系

老师在答辩时很容易追问“为什么这个结构能形成链”，所以这部分文档必须讲清楚。

## 十一、答辩时这一阶段可以怎么讲

如果老师问“为什么第二阶段先做基础区块链，而不是先做钱包和转账”，你可以这样回答：

1. 钱包和交易都是建立在区块链主干之上的
2. 没有稳定区块结构和链式存储，后续模块没有统一依附对象
3. 所以先实现区块和区块链，是为了先打通整个系统最基础的主链路

如果老师问“为什么现在就要引入数据库存储”，你可以这样回答：

1. 因为后续功能都依赖链的长期存在
2. 如果当前阶段只做内存链，后面 UTXO、钱包余额、网络同步都很难自然衔接
3. 所以持久化虽然不是最炫的功能，但它是后续扩展的关键基础

## 十二、本阶段完成后的下一步

Plan 2 完成后，就可以进入最简交易模型阶段。

也就是把现在区块中的简单数据字段，逐步过渡为更真实的交易结构。

到那时，系统已经有：

1. 区块
2. 区块链
3. 创世区块
4. 区块追加
5. 链遍历
6. 基础存储

这就为交易系统落地提供了直接基础。
