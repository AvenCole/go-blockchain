# Plan 17：脚本可视化与 GUI 演示增强

## 一、这一阶段要解决什么问题

Plan 16 已经把交易验证升级成了 Script / OP 虚拟机，但如果脚本能力只能在底层代码里存在，答辩时仍然不够直观。

Plan 17 的目标是：

1. 让 GUI 能直接展示脚本相关信息
2. 让钱包页面能解释“这个地址对应什么锁定脚本”
3. 让区块浏览页面能解释“这笔交易的输入输出脚本长什么样”
4. 同时修正 GUI 长时间持有链数据库句柄导致的潜在文件锁问题

## 二、为什么这个阶段要紧跟 Plan 16

因为脚本系统刚做完，如果不立刻把它转化成可视化演示能力，答辩时你还是得靠口头解释。  
而课程项目里，老师通常更容易通过界面理解：

1. `scriptSig` 在哪里
2. `scriptPubKey` 在哪里
3. 一个钱包地址是如何映射到标准 P2PKH 脚本的

所以这一阶段本质上是在把 Plan 16 的底层能力转化成可展示的上层能力。

## 三、本阶段范围

### 要做的内容

1. GUI 钱包视图增加标准锁定脚本展示
2. GUI 区块视图增加输入脚本和输出脚本展示
3. GUI 交易视图增加交易版本和是否启用 Script VM 标识
4. GUI 后端按请求打开/关闭链数据库，降低 Pebble 文件锁冲突风险
5. 同步文档

### 这一阶段不做的内容

1. 不新增复杂脚本类型
2. 不新增新的共识规则
3. 不重做整套 GUI 风格
4. 不实现 GUI 内脚本编辑器

## 四、理论讲解

### 1. 为什么脚本可视化很重要

脚本系统如果只在代码层生效，老师需要先理解你的数据结构和验证流程，才能明白你做了什么。  
但如果 GUI 里可以直接看到：

1. 钱包地址对应的 `scriptPubKey`
2. 交易输入中的 `scriptSig`
3. 交易输出中的 `scriptPubKey`

那很多原理就能在界面里直接讲清楚。

### 2. 为什么钱包页适合展示锁定脚本

因为钱包页天然就是“地址”的展示入口。  
而 P2PKH 的锁定脚本，本质上就是“这个地址的资金应当如何被解锁”的规则。  
所以在钱包页展示标准锁定脚本，非常符合老师的理解顺序。

### 3. 为什么区块页适合展示解锁脚本和锁定脚本

因为区块页本来就在展示交易详情。  
这里最适合说明：

1. 输入是如何解锁旧输出的
2. 输出又如何为下一次花费设置新规则

这正是 Script VM 最有教学价值的地方。

### 4. 为什么要顺手修正 GUI 数据库句柄问题

GUI 是展示层，不应该长期霸占底层 Pebble 句柄。  
如果 GUI 服务层一直持有数据库句柄，很容易在 Windows 上触发文件锁冲突，影响：

1. CLI 演示
2. 节点演示
3. GUI 自身稳定性

所以当前阶段顺便把 GUI 改成“按请求打开 / 使用 / 关闭”更合理。

## 六、实现步骤

### 第一步：补充 GUI 传输结构

为钱包、交易输入、交易输出增加脚本展示字段。

### 第二步：让后端视图层生成脚本字符串

后端把 `scriptSig`、`scriptPubKey`、交易版本和 Script VM 标识转换成前端可直接消费的文本和布尔值。

### 第三步：让前端页面展示脚本细节

钱包页展示标准锁定脚本，区块页展示输入和输出脚本。

### 第四步：清理 GUI 数据库句柄生命周期

把 GUI 后端改成短生命周期数据库访问，减少文件锁问题。

## 七、验收标准

1. 钱包页能显示标准 P2PKH 锁定脚本
2. 区块页能显示 `scriptSig`
3. 区块页能显示 `scriptPubKey`
4. 区块页能显示交易版本和 Script VM 状态
5. GUI 后端不再长期缓存链数据库句柄
6. Go、前端、Wails 构建通过
7. 文档同步更新

## 八、常见错误

### 错误 1：脚本只在 CLI 可见，GUI 看不到

这样会浪费 GUI 的演示价值。

### 错误 2：前端直接拼装脚本逻辑

脚本展示应由后端视图层整理好，前端负责展示，而不是把链逻辑搬到页面里。

### 错误 3：为了显示脚本而引入新的高耦合层

当前最合理的方式是扩展已有 GUI ViewModel，不需要另外造复杂中间层。

## 九、答辩时可以怎么讲

你可以这样解释：

1. Plan 16 完成了脚本系统本身
2. Plan 17 把脚本系统变成了可视化演示能力
3. 老师现在不仅能看到地址和余额，还能看到地址对应的标准锁定脚本
4. 在区块详情里，也可以看到输入脚本与输出脚本的衔接关系

## 十、本阶段完成后的下一步

Plan 17 完成后，更适合继续往高分方向扩展的内容包括：

1. 更完整的最长链 / 分叉切换
2. 更复杂的脚本类型
3. 更强的 GUI 网络状态与运维面板
4. 更接近 Bitcoin Core 的多窗口信息组织方式
