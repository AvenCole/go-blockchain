# Plan 7：UTXO 缓存与查询优化

## 一、这一阶段要解决什么问题

Plan 6 已经把交易做成了签名 UTXO 模型，但当前余额查询和找可花费输出的过程仍然可能频繁依赖链上数据遍历。

Plan 7 的目标是：

1. 单独维护一个 UTXO 集缓存
2. 让余额查询更快
3. 让查找可花费输出更快
4. 支持在缓存出问题时重建索引

## 二、为什么这一阶段必须单独做

因为随着区块和交易数量变多，如果每次：

1. 查询余额
2. 构造交易
3. 查找未花费输出

都去全链扫描，性能会越来越差。

Plan 7 的作用就是把“正确性已成立”的 UTXO 模型进一步优化成“查询更快、可维护、可恢复”的结构。

## 三、本阶段范围

### 要做的内容

1. 为 UTXO 集定义缓存存储形式
2. 把 UTXO 缓存写入 Pebble
3. 让 `getbalance` 优先走缓存
4. 让 `send` 优先走缓存找可花费输出
5. 在区块追加后增量更新缓存
6. 提供 `ReindexUTXO`
7. 提供 CLI `reindexutxo`
8. 同步文档

### 这一阶段不做的内容

1. 不做 Merkle 树
2. 不做 PoW
3. 不做网络同步
4. 不做交易池

## 四、核心理论

### 1. UTXO 集为什么适合缓存

UTXO 本质上就是“当前仍能被花费的输出集合”。  
这个集合会随着每个新区块增加和删除部分记录，非常适合作为单独状态缓存维护。

### 2. 为什么缓存要支持重建

缓存不是链本身。  
如果缓存损坏、丢失或者逻辑升级，就必须能够从完整链重新扫描恢复。

所以重建索引能力是这类缓存系统的必要部分。

### 3. 为什么缓存和主链数据仍然可以共存在同一个 Pebble 中

因为 Pebble 本身就是 KV 存储，完全可以通过不同 key 前缀把：

1. 区块数据
2. 链尾元数据
3. UTXO 缓存

分开管理。

这样既避免引入新存储系统，也方便后续维护。

## 六、推荐设计

### 1. 缓存键设计

推荐使用单独前缀，例如：

1. `utxo-<txid>`

值可以保存该交易当前仍未花费的输出列表。

### 2. 两种更新方式

当前阶段建议都实现：

1. **全量重建**：从完整链扫描生成 UTXO 缓存
2. **增量更新**：新区块入链后，只更新受影响的 UTXO 项

### 3. 查询优先级

从 Plan 7 开始：

1. `getbalance` 优先走缓存
2. `send` 选择可花费输出优先走缓存

## 七、开发落地顺序

### 第一步：定义缓存键前缀

把 UTXO 缓存从普通区块键中分离出来。

### 第二步：实现全量重建

从链上完整扫描所有交易，重新得到当前 UTXO 集。

### 第三步：实现增量更新

在新区块成功入链后：

1. 删除已花费输出
2. 加入新生成输出

### 第四步：切换查询逻辑

让余额和可花费输出查询优先使用缓存。

### 第五步：接入 CLI

提供：

1. `reindexutxo`

## 八、CLI 如何演示

建议演示顺序：

1. `createwallet`
2. `createwallet`
3. `createblockchain <miner-address>`
4. `send <miner-address> <alice-address> 20`
5. `getbalance <miner-address>`
6. `getbalance <alice-address>`
7. `reindexutxo`
8. 再次执行 `getbalance`

这样老师能看到：

1. UTXO 缓存在正常工作
2. 重建后结果不变
3. 缓存是可以恢复的，不是脆弱黑盒

## 九、验收标准

1. UTXO 缓存已建立
2. send 走缓存查找可花费输出
3. getbalance 走缓存
4. reindex 能成功执行
5. 重建前后余额结果一致
6. 测试通过
7. 文档同步更新

## 十、常见错误

### 错误 1：只做缓存，不做重建

这样缓存一旦损坏，系统就不可恢复。

### 错误 2：缓存更新和链更新不同步

这样余额和交易会出现不一致。

### 错误 3：把缓存做成另一套和链无关的逻辑

UTXO 缓存本质上是链状态的派生结果，不能脱离链独立演化。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段已经先保证交易模型正确
2. 这一阶段开始优化 UTXO 查询性能
3. 通过单独维护 UTXO 集缓存，避免每次都全链扫描
4. 但同时保留重建能力，保证缓存不是单点风险
