# Plan 19：链重组后的 Mempool 恢复策略

## 一、这一阶段要解决什么问题

Plan 18 已经让系统支持：

1. 保存侧链块
2. 在更长分叉出现时切换到最长链
3. 切换后重建 UTXO

但还有一个非常关键的问题没有处理完整：

1. 如果旧主链上的某笔普通交易因为链切换而“掉下来了”
2. 并且这笔交易在新主链下依然合法
3. 那么它不应该直接消失，而应该回到 Mempool 等待重新打包

Plan 19 的目标，就是补齐这一层链重组后的交易恢复策略。

## 二、为什么这个阶段必须做

如果系统只能切换主链，但不能恢复被重组掉的交易，那么会出现一个很不合理的现象：

1. 用户之前已经成功发送并出块的交易
2. 因为更长分叉出现而不在主链上了
3. 但系统没有把它重新放回待打包队列
4. 用户就会感觉“交易凭空消失了”

这不仅不符合区块链系统的基本直觉，也会让答辩时老师进一步追问“那旧链上的交易去哪了”。

## 三、本阶段范围

### 要做的内容

1. 在最长链切换时识别旧主链独有的普通交易
2. 识别新主链上已经确认的普通交易
3. 对旧主链独有、且在新主链未确认的交易尝试重新放回 Mempool
4. 对新主链中已确认的交易从 Mempool 中移除
5. 提供 CLI 演示入口 `simreorg`
6. 补测试与文档

### 这一阶段不做的内容

1. 不实现复杂 orphan transaction 池
2. 不实现完整比特币级别的重组后依赖交易拓扑恢复
3. 不实现网络级别的重组通知广播
4. 不恢复已经在新主链下失效的交易

## 四、核心理论讲解

### 1. 什么是链重组后的交易恢复

链重组发生后，原主链上的某些块会退出主链。  
这些块里的普通交易，如果：

1. 不在新主链中
2. 在新主链账本状态下仍然合法

那就应该重新回到 Mempool，等待后续矿工再次打包。

### 2. 为什么不是所有旧交易都要恢复

因为旧主链掉下来的交易有几种情况：

1. 有些已经在新主链里重新确认了，不需要恢复
2. 有些因为新主链状态变化已经失效，也不应该恢复
3. 只有那些“仍然合法、但当前未确认”的交易，才适合重新放回 Mempool

### 3. 为什么恢复顺序很重要

旧主链上的交易如果有先后依赖关系，那么恢复时的顺序也可能影响能否重新加入池中。  
当前课程项目里，为了保持实现可讲解、可控，采用的是：

1. 按旧主链分支从靠近公共祖先到旧 tip 的顺序尝试恢复
2. 依赖当前链状态和现有 Mempool 校验规则
3. 只恢复当前规则下能重新成立的交易

这是一种教学型、稳妥的策略。

### 4. 为什么要顺便清理已在新主链确认的池中交易

如果链切换后，Mempool 里还保留着已经被新主链确认的交易，那池中状态就会和主链状态冲突。  
所以重组后需要两步同时做：

1. 删掉已在新主链确认的池中交易
2. 恢复旧主链掉下来的、仍然合法的交易

## 五、代码结构讲解

本阶段主要涉及：

1. `internal/blockchain/blockchain.go`
2. `internal/blockchain/blockchain_test.go`
3. `internal/cli/app.go`
4. `internal/cli/app_test.go`
5. `docs/plan/plan19.md`

### 1. `blockchain.go`

这里负责：

1. 在链切换时识别新旧分支差异
2. 生成“新主链已确认交易集合”
3. 生成“可尝试恢复的旧分支交易列表”
4. 重建 UTXO 后对 Mempool 做清理和恢复

### 2. `app.go`

这里增加 `simreorg`，用于显式演示：

1. 一笔交易先进入旧主链并被确认
2. 更长分叉出现
3. 链切换后该交易被恢复回 Mempool
4. 对应收款余额回退到新主链视图

## 六、实现步骤

### 第一步：在切换前识别新旧分支差异

先找出：

1. 哪些交易已经在新主链中确认
2. 哪些交易只存在于旧主链独有区段

### 第二步：切换 tip 并重建 UTXO

先保证账本状态切到新主链，再考虑 Mempool 恢复。

### 第三步：清理和恢复 Mempool

1. 删除池中已在新主链确认的交易
2. 把旧主链掉下来的、仍然合法的交易尝试重新放回池中

### 第四步：增加 CLI 演示入口与测试

让这一行为能稳定复现，而不是只存在于内部逻辑中。

## 七、CLI 如何演示

建议演示顺序：

1. `createwallet`
2. `createwallet`
3. `createblockchain <miner-address>`
4. `simreorg <miner-address> <receiver-address> 20 1`
5. `printmempool`
6. `getbalance <receiver-address>`

演示重点：

1. 交易先被打进旧主链
2. 更长分叉出现后，主链切换
3. 原交易重新回到 Mempool
4. 收款人余额按新主链视角回退

## 八、验收标准

1. 最长链切换后会清理已确认池中交易
2. 最长链切换后会恢复旧主链掉下来的合法交易
3. `simreorg` 可稳定演示
4. 测试通过
5. 文档同步更新

## 九、常见错误

### 错误 1：只切换 tip，不处理旧主链交易

这会让交易在链重组后无故消失。

### 错误 2：不区分“新主链已确认交易”和“旧主链掉下来的交易”

这样容易把不该恢复的交易错误放回 Mempool。

### 错误 3：在切换前直接恢复交易

恢复动作应当基于新主链状态，否则校验基础就错了。

## 十、答辩时可以怎么讲

你可以这样解释：

1. Plan 18 解决的是“链怎么切换”
2. Plan 19 解决的是“链切换后交易怎么办”
3. 系统现在不仅会选择最长链，还会尽量保留那些在新主链下仍然有效的用户交易
