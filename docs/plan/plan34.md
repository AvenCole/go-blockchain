# Plan 34：GUI 分叉 / 重组一键演示

## 一、这一阶段要解决什么问题

到 Plan 33 为止，GUI 已经可以一键演示：

1. 双节点启动
2. 节点同步
3. 交易广播
4. 出块同步

但这条链路展示的是“正常路径”。  
如果想在答辩里拿更高分，只展示正常路径还不够，因为老师往往会继续追问：

1. 你有没有做分叉处理
2. 你怎么证明最长链切换真的发生
3. 链重组后，被旧主链确认的交易会怎样
4. 这些行为能不能在界面里直接看到

Plan 34 的目标，就是把“已确认交易被更长分叉替换，并重新回到 Mempool”的过程，做成 GUI 可一键触发、可重复验证、可直接讲解的演示场景。

## 二、为什么这一阶段值得做

因为区块链课程里，分叉和重组通常是区分“只做了普通链结构”与“真正理解共识切换”的关键点。

如果你只能说：

1. 系统支持 longest-chain
2. 代码里写了 reorg

老师仍然可能觉得这是纸面能力。  
但如果你能在 GUI 里直接演示：

1. 一笔交易先被打包确认
2. 随后出现更长分叉
3. 原确认块被替换
4. 交易重新回到 Mempool
5. peer 节点也同步到新的链状态

那这个系统的完成度和可解释性会明显更强。

## 三、本阶段范围

### 要做的内容

1. 抽出可复用的重组演示服务逻辑，避免 CLI 和 GUI 各写一套
2. 为 GUI 网络页增加“一键分叉 / 重组演示”
3. 先让双节点共享同一条已确认交易所在主链
4. 再向主节点注入更长竞争分叉，触发链重组
5. 再让 peer 节点重新同步并观察它的重组结果
6. 把重组摘要结构化返回给前端
7. 在节点状态里补充每个节点的最近重组状态
8. 为 GUI 控制台补充 `runreorgdemo`
9. 补充自动化测试
10. 同步本计划文档

### 这一阶段不做的内容

1. 不做真实 P2P 断网 / 复网图形化时间轴
2. 不做多种竞争分叉随机生成器
3. 不做跨多条平行分叉的复杂可视化动画
4. 不做节点自动回收与演示场景清理器

## 四、核心理论讲解

### 1. 为什么要强调“已确认交易被回滚”

因为这正是链重组最容易被老师理解的地方。

普通展示“高度从 1 变 2”并不能说明问题。  
真正能说明问题的是：

1. 某笔交易原本已经在主链区块里
2. 旧主链被更长新链替代
3. 旧块里的普通交易不再被当前主链确认
4. 系统把它们重新放回 Mempool，等待再次打包

这说明你不仅做了 longest-chain，还处理了链切换后的账本与交易池一致性。

### 2. 为什么这次仍然采用“一键场景”

因为重组演示步骤比普通同步更长：

1. 启动节点
2. 初始化创世链
3. 同步创世块
4. 发送交易
5. 挖出旧主链块
6. peer 同步旧主链
7. 构造更长竞争分叉
8. 触发重组
9. 再次同步 peer
10. 检查 Mempool 恢复与重组状态

如果让你手动一点点做，现场非常容易乱。  
所以这里继续采用“服务层编排 + GUI 一键触发”的策略，把复杂链路固定成标准答辩脚本。

### 3. 为什么要把 CLI 的重组逻辑提炼成独立模块

因为 CLI 和 GUI 都需要同一类“演示级重组编排”能力。

如果两边各自复制：

1. 后续改一边容易忘另一边
2. 演示行为会不一致
3. 测试覆盖会分裂

所以这一阶段把重组演示逻辑提炼到独立的 `internal/demo/` 中，让：

1. CLI 的 `simreorg`
2. GUI 的重组演示

都调用同一套底层实现。

这就是“降低耦合、提高复用”的最佳实践。

### 4. 为什么节点状态里要增加 `LastReorg`

因为仅仅返回总演示结果还不够。

答辩时老师常常会继续问：

1. 哪个节点发生了重组
2. 它原来多高，后来多高
3. peer 是否也真的切换过去了

如果节点面板里直接显示最近一次重组摘要，这些问题就不需要靠口头补充，而是可以直接看界面。

## 六、实现步骤

### 第一步：抽出可复用重组演示逻辑

先把 CLI 原来内嵌的 `simreorg` 逻辑提炼到 `internal/demo/reorg.go`。

这样做的好处：

1. CLI 不再独占这套实验逻辑
2. GUI 可以直接复用
3. 测试更集中
4. 后续扩展更容易

### 第二步：让节点状态能读取最近一次重组

在 `network.Node` 中增加读取最近重组状态的方法，再由 GUI 的 `Nodes()` 把它转换成前端可用结构。

这样网络页不只是“看高度”，而是真正能看节点链状态的变化历史。

### 第三步：实现 `RunNetworkReorgDemo()`

这一步是本阶段的核心。

内部顺序是：

1. 准备两个钱包
2. 启动 source 节点和 peer 节点
3. 初始化 source 链
4. 让 peer 同步到创世块
5. 通过 source 节点发送交易并挖矿
6. 等待 peer 同步到旧主链高度
7. 对 source 本地链注入更长竞争分叉
8. 触发 source 本地 reorg
9. 再让 peer 重新连接 source，同步到新主链
10. 检查交易恢复与 peer 重组结果

### 第四步：把能力接到 GUI 和控制台

统一通过：

1. `app.go` 暴露给 Wails
2. `frontend/src/api/backend.ts` 做前端调用封装
3. `NetworkPage.tsx` 做图形化按钮
4. `ConsolePage.tsx` 支持 `runreorgdemo`

这样既可以点按钮演示，也可以在 GUI 内置终端演示。

## 七、关键代码讲解

### 1. `RunReorgMempoolRecovery()`

这个函数面向“完整重组实验”：

1. 先发交易
2. 再挖矿确认
3. 然后转去执行更长分叉替换

它适合 CLI 的 `simreorg` 命令，因为 CLI 需要从零跑完整实验。

### 2. `ForceReorgToLongerGenesisFork()`

这个函数面向“已经有一条旧主链时，强制切到更长分叉”的场景。

GUI 重组 demo 要先让 peer 看见旧主链，因此更适合调用这个函数。  
这样可以把“先确认旧链”和“后触发重组”拆开，更灵活。

### 3. `RunNetworkReorgDemo()`

这是本阶段的 GUI 编排核心。

它把：

1. 节点网络同步
2. 已确认交易
3. 链重组
4. peer 重新同步

放在同一个流程里跑完。  
这样老师能在一个按钮里看到系统对“异常但合理的链行为”的处理能力。

### 4. `Node.LastReorgStatus()`

这个函数的价值不在于“多读一个状态”，而在于它保证了读取过程走节点自己的链访问路径，不额外打破锁顺序。

这对于 Windows + Pebble 这样的文件锁场景尤其重要。  
也就是说，这里不仅是在补功能，也是在补稳定性。

## 八、常见错误

### 错误 1：只在主节点本地重组，但没有让 peer 重新同步

这样演示出来的只是“单机链状态切换”，不是网络里的链切换。

### 错误 2：重组后只看高度，不检查交易是否恢复

这样证明不了 Mempool 恢复策略真的正确。

### 错误 3：GUI 和 CLI 的重组演示用两套独立逻辑

这样后续非常容易出现：

1. 一边改了，另一边没改
2. 演示结果不一致
3. 测试无法共享

### 错误 4：节点状态读取不考虑 Pebble 句柄冲突

Windows 下很容易出现“文件被其他进程占用”的问题。  
所以读取节点链状态时，必须尽量走统一的节点访问路径。

## 九、验收标准

1. GUI 网络页新增“一键分叉 / 重组演示”入口

3. 结果中可以看到交易是否恢复、源节点高度变化、peer 是否完成重组
4. 节点状态卡片可以看到 `last reorg` 摘要
5. GUI 控制台支持 `runreorgdemo`
6. CLI 的 `simreorg` 仍然可用
7. `go test ./...`、`go build ./...`、前端类型检查与构建通过

## 十、答辩时可以怎么讲

你可以这样解释：

1. Plan 33 证明了网络同步和正常出块链路
2. Plan 34 进一步证明系统能处理更长分叉带来的链重组
3. 不只是切换 tip，还会把旧主链里的普通交易恢复回 Mempool
4. 而且这个过程不仅发生在 source 节点，也会传播到 peer 节点

这会让老师很容易理解：你的系统不是只会“跑通 happy path”，而是真的考虑了区块链里的关键链切换行为。
