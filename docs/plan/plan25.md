# Plan 25：更细粒度的网络事件观测

## 一、这一阶段要解决什么问题

到 Plan 24 为止，系统已经有链事件面板，但网络层自身的运行细节仍然不够可见。  
比如在答辩时，老师可能继续追问：

1. 节点什么时候建立监听
2. 什么时候发现了 peer
3. 什么时候发送了 version
4. 孤儿块是何时被缓存、何时被补回的

如果这些都只能靠你口头解释，演示层还是不够强。

Plan 25 的目标，就是让 GUI 托管节点具备最近网络事件观测能力。

## 二、为什么这一阶段值得做

因为现在项目已经具备：

1. GUI 网络管理
2. 孤儿块缓冲
3. 链切换与重组事件

这时再补充网络事件观测，老师就能同时看到：

1. 链发生了什么
2. 节点网络层发生了什么

这会让整个系统从“功能能跑”进一步提升到“行为可解释”。

## 三、本阶段范围

### 要做的内容

1. 为节点增加最近网络事件列表
2. 记录监听启动、peer 加入、version 发送、交易接收、区块导入、孤儿块缓存、父块补取、孤儿块恢复等事件
3. 通过 GUI 服务层把最近网络事件暴露给前端
4. 在 GUI 网络页中展示每个节点的最近事件摘要
5. 增加测试验证事件确实被记录
6. 同步文档

### 这一阶段不做的内容

1. 不实现完整分布式 tracing 系统
2. 不实现持久化网络事件数据库
3. 不记录所有 socket 级别细节
4. 不做复杂图表和实时曲线

## 四、核心理论讲解

### 1. 为什么链事件和网络事件要区分

链事件描述的是：

1. 区块链状态如何变化

网络事件描述的是：

1. 节点是如何收到这些变化的
2. 在收到之前做了哪些网络动作

两者互补，不能互相替代。

### 2. 为什么节点事件要做成最近列表

和链事件一样，课程项目里最重要的是：

1. 能看最近发生了什么
2. 能解释关键行为
3. 不把系统做得过重

所以用一个固定长度的最近事件列表，比完整日志系统更适合当前目标。

### 3. 为什么孤儿块相关事件尤其值得展示

因为孤儿块缓冲本身是一个“看不到就很难证明存在”的能力。  
一旦最近网络事件里能直接看到：

1. orphan buffered
2. parent requested
3. orphan resolved

那答辩时说服力会非常高。

## 六、实现步骤

### 第一步：给节点增加最近事件缓冲

先让节点能保存固定数量的最近网络事件。

### 第二步：在关键网络动作里记事件

把最值得演示的行为记录下来，例如：

1. listen
2. peer
3. connect
4. block_import
5. orphan
6. parent_request
7. orphan_resolved

### 第三步：通过 GUI 服务层暴露事件

把节点最近事件整理成前端能直接展示的结构。

### 第四步：在网络页展示最近事件

让每个节点状态卡里不仅有 peer、高度、orphans，还有最近事件摘要。

## 七、验收标准

1. 节点能记录最近网络事件
2. 孤儿块缓冲与恢复会留下事件记录
3. GUI 网络页能显示每个节点最近事件
4. 测试通过
5. 文档同步更新

## 八、常见错误

### 错误 1：只记录链事件，不记录网络动作

这样仍然解释不了节点层面发生了什么。

### 错误 2：记录过多无关事件

这样反而会淹没关键行为，降低答辩可读性。

### 错误 3：最近事件只在内存里但没有通过 GUI 暴露

那它仍然不利于展示。

## 九、答辩时可以怎么讲

你可以这样解释：

1. 前面阶段解决了链行为可视化
2. Plan 25 进一步把节点网络行为也变得可视化
3. 现在不仅能看到链变了什么，还能看到节点为了处理这些变化做了什么

## 十、本阶段完成后的下一步

Plan 25 完成后，更适合继续推进的方向包括：

1. 更复杂的脚本类型
2. 更接近 Bitcoin Core 的多窗口信息组织

4. 更细粒度的脚本与网络联合演示面板
