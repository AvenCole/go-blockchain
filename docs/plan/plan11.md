# Plan 11：网络模拟

## 一、这一阶段要解决什么问题

前面的阶段已经让单节点链路基本跑通，但区块链系统如果只有一个进程，就还不算真正的“链网络”。

Plan 11 的目标是：

1. 让节点可以启动
2. 让节点能够发现彼此
3. 让交易可以广播
4. 让新区块可以广播
5. 让落后节点可以同步更长链

## 二、为什么网络模拟必须单独做成一个阶段

因为它不再是单机数据结构问题，而是分布式行为问题。

这一阶段开始回答：

1. 节点之间怎么通信
2. 节点怎么知道其他节点存在
3. 新交易和新区块怎么传播
4. 新节点怎么追上已有链状态

## 三、本阶段范围

### 要做的内容

1. 定义网络消息结构
2. 定义 `Node`
3. 实现 TCP 监听
4. 实现版本握手
5. 实现已知节点列表同步
6. 实现区块同步
7. 实现交易广播
8. 实现区块广播
9. 增加 `startnode`
10. 同步文档

### 这一阶段不做的内容

1. 不做真实公网 P2P
2. 不做 Gossip 优化
3. 不做复杂分叉处理
4. 不做交易优先级同步策略

## 四、核心理论

### 1. 节点发现是什么

节点发现就是让一个节点知道“还有哪些节点存在”。

### 2. 为什么先做本地 TCP 模拟

因为课程项目更强调流程正确和可演示。  
用本地 TCP 就足够把：

1. 启动
2. 握手
3. 广播
4. 同步

这一整套链路讲清楚。

### 3. 为什么区块同步可以先从“线性扩展”开始

完整的最长链分叉处理可以放到后续进一步增强。  
本阶段先保证“一个节点出块，另一个节点能跟上”即可。

## 六、推荐设计

### 1. 消息类型

建议至少有：

1. `version`
2. `addr`
3. `getblocks`
4. `blocks`
5. `tx`
6. `block`

### 2. 节点结构

至少包含：

1. 节点地址
2. 数据目录
3. 已知节点列表
4. 可选矿工地址

### 3. 同步策略

先做“高度比较 + 缺块补齐”的基础同步。

## 七、开发落地顺序

### 第一步：定义消息格式

先把网络消息明确下来。

### 第二步：实现节点监听

节点能启动并接收连接。

### 第三步：实现版本握手和节点发现

让节点之间先知道对方存在以及各自高度。

### 第四步：实现区块同步

新节点能够补到已有链高度。

### 第五步：实现交易广播

一个节点发出的交易可以进入另一个节点的 mempool。

### 第六步：实现区块广播

一个节点挖出的新区块可以被另一个节点导入。

## 八、CLI 如何演示

建议演示方式：

1. 开两个终端
2. 一个 `startnode <addr>`
3. 另一个 `startnode <addr> <seed>`
4. 让主节点创建交易并挖矿
5. 验证从节点链高度同步变化

## 九、验收标准

1. 两个节点都能启动
2. 节点可互相发现
3. 区块可同步
4. 交易可广播
5. 新块可广播
6. 测试通过
7. 文档同步更新

## 十、常见错误

### 错误 1：只做消息结构，不做真实监听

这样不能算网络模拟落地。

### 错误 2：节点发现和同步是两套分裂逻辑

这会让流程很难解释。

### 错误 3：同步不做区块合法性校验

这样网络节点会变成“谁发什么就信什么”，和前面阶段的安全设计冲突。

## 十一、答辩时可以怎么讲

你可以这样解释：

1. 单机链路已经完成
2. 这一阶段开始把单机链路扩展成多节点模拟
3. 重点不是公网部署，而是把“发现、广播、同步”这三条主链路跑通
