# Plan 39：网络事件筛选与搜索工作台

## 一、这一阶段要解决什么问题

到 Plan 38 为止，网络页和控制台已经具备了比较完整的操作能力：

1. 网络页可以看拓扑、时间线、链事件、节点焦点
2. 控制台可以直接执行命令，也可以用预设命令
3. 多节点、多流程演示已经比较完整

但随着系统事件越来越多，又出现了一个新的问题：

1. 时间线会不断堆积
2. 链事件列表会越来越长
3. 单节点最近网络事件也会越来越多
4. 用户很难快速定位“某一类事件”或“某个节点刚刚做了什么”

这会让网络页再次慢慢退化成信息堆叠，而不是一个真正好用的运维工具。

Plan 39 的目标，就是把网络页里的事件区域升级成可筛选、可搜索的工作台：

1. 时间线支持按类型、来源、关键字筛选
2. 链事件支持按类型、关键字筛选
3. 焦点节点最近事件支持按类型、关键字筛选
4. 所有筛选区都显示“当前匹配数 / 总数”

## 二、为什么这一阶段值得做

工具类界面随着信息量增大，最先出现的问题通常不是“没有信息”，而是“信息太多以后不好找”。

例如网络页里，用户可能只关心：

1. `reorg`
2. `orphan`
3. `tip_announce`
4. 某个节点最近的 `connect`、`block_import`

如果没有筛选能力，用户就只能靠肉眼滚动查找。  
这会直接降低网络页作为工具的使用价值。

所以本阶段不是增加新的底层区块链能力，而是在已有观测能力之上补齐“检索效率”。

## 四、核心理论讲解

### 1. 为什么事件系统必须补“筛选层”

观测系统的价值不仅在于“记录了什么”，还在于“能不能快速找到想看的内容”。

如果系统只能把事件全部堆出来，那么信息量越大，实际可用性越差。  
所以筛选层是观测系统从“展示信息”走向“支持操作”的关键一步。

### 2. 为什么本阶段优先做前端筛选

当前网络页里的事件量还属于桌面工具可直接处理的规模。  
此时最合理的做法是：

1. 继续复用已有后端返回的数据
2. 在前端本地完成筛选和搜索

这样做的好处是：

1. 不需要引入新的后端接口
2. 不会增加后端耦合
3. 实现速度快，交互响应也直接

### 3. 为什么要把筛选条做成复用组件

这一阶段里，需要筛选的区域有三处：

1. 时间线
2. 链事件
3. 焦点节点最近事件

如果每处都手写一套搜索框和下拉框，很快就会出现：

1. 代码重复
2. 样式不一致
3. 后续维护困难

所以本阶段增加统一的 `EventFilterToolbar` 组件，负责承载：

1. 搜索
2. 类型筛选
3. 来源筛选（可选）
4. 当前匹配统计

### 4. 为什么要抽离事件筛选工具函数

筛选逻辑本质上属于数据处理，不应该写死在 JSX 里。  
所以本阶段把逻辑放入 `networkView.ts`：

1. 收集可选事件类型
2. 收集时间线来源
3. 文本匹配
4. 链事件筛选
5. 节点事件筛选
6. 时间线筛选

这样组件本身就只负责展示与交互，符合低耦合原则。

## 七、关键代码讲解

### 1. `collectKinds()`

这个函数负责从任意事件数组里提取唯一事件类型。  
它解决的是“下拉筛选项从哪里来”的问题。

### 2. `collectSources()`

这个函数用于时间线来源筛选。  
时间线既包含主链事件，也包含节点事件，所以来源维度需要单独提取。

### 3. `matchesQuery()`

这是文本搜索的核心。  
它会把：

1. 事件类型
2. 时间戳
3. 来源
4. 详情 / 摘要

组合成可搜索文本，统一做小写匹配。

### 4. 焦点节点切换时重置筛选

节点焦点变化后，如果继续保留旧搜索条件，很容易出现“明明有事件却看不到”的误判。  
所以本阶段在焦点节点变化时重置节点事件筛选条件，让交互更符合直觉。

## 八、常见错误

### 错误 1：只加搜索框，不加类型筛选

这样能解决一部分问题，但用户仍然很难快速定位特定事件类别。

### 错误 2：在每个组件里复制粘贴筛选逻辑

这样很快会导致逻辑不一致，也不利于后续维护。

### 错误 3：先截断列表再筛选

这样会让更早但符合条件的事件永远搜不到。  
正确顺序应该是先筛选，再决定展示多少条。

### 错误 4：切换焦点节点时不处理旧筛选条件

这样会让用户误以为新节点没有事件。
