# Plan 1：项目初始化

## 一、这一阶段要解决什么问题

Plan 1 的目标不是立刻把区块链做出来，而是先把整个项目的基础骨架搭好。

如果这个阶段做得不清楚，后面在实现区块、钱包、交易、PoW、网络、GUI 的时候，就会不断出现这些问题：

1. 目录乱
2. 命名不统一
3. 配置入口分散
4. CLI 没有统一入口
5. 文档跟不上代码
6. 到答辩时很难解释为什么项目是这样组织的

所以，Plan 1 的本质是：先把项目做成一个“可以继续长大”的骨架。

## 二、为什么初始化不能只随便建几个文件夹

很多人做课程项目时，会先写一个 `main.go`，然后想到什么写什么。短期看很快，长期看很容易返工。

这个项目后续会逐步加入：

1. 区块链结构
2. 钱包
3. 交易与签名
4. UTXO
5. Merkle 树
6. PoW
7. 交易池
8. 网络模拟
9. 安全攻击模拟
10. GUI

这些模块之间是连续演进关系，不是彼此独立的小功能。因此初始化阶段就要先想清楚：

1. 后续代码往哪里放
2. 程序怎么启动
3. 参数从哪里读
4. 文档如何和开发同步

## 三、本阶段范围

### 要做的内容

1. 初始化 Go module
2. 建立目录结构
3. 建立主程序入口
4. 建立最小 CLI 主干
5. 建立配置层
6. 明确默认技术规范
7. 建立最小测试规则
8. 建立 `docs/plan/` 文档体系

### 这一阶段不做的内容

1. 不实现区块链核心逻辑
2. 不实现钱包和地址生成
3. 不实现真实交易系统
4. 不实现 PoW
5. 不实现网络广播
6. 不实现 GUI 页面

这样划分的好处是：初始化阶段目标非常明确，不会越做越散。

## 四、目录结构应该怎么理解

推荐的初始化目录结构如下：

```text
cmd/
  go-blockchain/
internal/
  blockchain/
  cli/
  config/
  wallet/
  network/
  gui/
data/
docs/
  plan/
tests/
```

### 1. `cmd/go-blockchain/`

这里放程序入口，也就是 `main.go`。

为什么入口要单独放在 `cmd/` 下面？

因为 Go 项目通常会把真正的启动程序放在 `cmd/<app-name>/`，这样结构清晰，也方便以后扩展多个入口程序。

### 2. `internal/blockchain/`

虽然 Plan 1 还不会实现真正的区块链逻辑，但后续整个项目的主线都围绕它展开，所以这个目录要尽早确定。

### 3. `internal/cli/`

这个目录负责命令行入口逻辑。

后续像 `createblockchain`、`send`、`getbalance`、`startnode` 之类命令，都应该从这个层进入，而不是全部堆在 `main.go`。

### 4. `internal/config/`

这个目录负责统一配置。

如果不先把配置入口统一，后面节点端口、数据路径、日志等级这些参数就会散落在各个文件里，维护会很痛苦。

### 5. `internal/wallet/`、`internal/network/`、`internal/gui/`

这些目录在 Plan 1 中可以只做最小预留，不需要写复杂实现。

这样做的目的不是装样子，而是让项目从一开始就和主流程保持一致。

### 6. `data/`

后续链数据、UTXO 数据、钱包数据等都需要落盘，这个目录用于统一管理本地数据。

### 7. `docs/plan/`

这个目录是整个项目开发过程的解释层。

你后面每推进一个阶段，我都要同步把这个阶段的文档补进来，不会只给建议不落地。

### 8. `tests/`

这个目录不是用来放所有测试。

Go 项目的单元测试优先放在对应包目录下，用 `_test.go`。  
`tests/` 主要留给后续集成测试、验收脚本、演示校验脚本。

## 五、为什么不建议提前创建 `core` 和 `utils`

看起来 `core` 和 `utils` 很方便，但对这个项目来说，初始化阶段反而容易埋坑。

原因有三个：

1. `core` 很容易变成什么都往里放的大杂烩
2. `utils` 很容易变成垃圾桶目录
3. 这个项目的主线本来就很清楚，按领域拆分更自然

所以当前阶段更适合按“区块链、钱包、网络、GUI、CLI、配置”这些真实领域来组织。

## 六、最小 CLI 应该做到什么程度

Plan 1 的 CLI 只要求建立主干，不要求业务功能完整。

建议做到下面几项：

1. `--help` 可以输出帮助信息
2. `version` 或 `about` 可以输出项目信息
3. `doctor` 占位命令可以被识别

这里选择 `doctor` 作为占位命令，是因为它比较中性，不会和后续真实业务命令冲突，而且很适合后续扩展为环境检查、配置检查、数据目录检查等功能。

## 七、配置层为什么现在就要做

后面这个项目会逐步涉及：

1. 数据库存储位置
2. 节点端口
3. 是否启用网络模式
4. 日志输出
5. 演示模式或本地模式

如果这些内容从一开始没有统一配置入口，后面就只能到处找硬编码。

所以初始化阶段就应该至少统一下面这些默认配置：

1. 项目名
2. 数据目录
3. 默认端口预留
4. 日志级别
5. 网络模式预留

## 八、默认技术规范是什么意思

Plan 1 会先写一份“默认技术规范”，它的意思不是“以后绝对不能改”，而是：

1. 当前先统一一个方向
2. 后面实现时如果有更强理由，可以在不破坏整体结构的前提下调整

当前推荐写成这样的规则：

1. 哈希默认使用 SHA-256
2. 调试展示默认使用 JSON
3. 哈希和签名时使用稳定字节编码，不直接拿调试 JSON 去做密码学输入
4. 后续签名方案默认使用 ECDSA
5. 地址方案先记录方向，不在 Plan 1 完整实现

这样做的好处是：

1. 你后面写代码时不会完全没有方向
2. 答辩时也能解释为什么一开始这样定
3. 又不会因为过早锁死细节而难以调整

## 九、这一阶段的开发落地顺序

### 第一步：初始化 Go module

先让仓库变成标准 Go 项目，这是后续所有构建、测试、运行的前提。

### 第二步：建立目录骨架

先把顶层目录和内部领域目录建立好，但不要铺大量空实现文件。

### 第三步：建立 `main.go`

让项目先有启动入口，至少能执行并输出基本信息。

### 第四步：建立 CLI 主干

让程序能解析基础命令，而不是所有逻辑都写在 `main.go` 里。

### 第五步：建立配置层

让后续所有模块都从统一入口读取配置。

### 第六步：建立测试骨架

至少先保证配置和 CLI 是可验证的。

### 第七步：补 Plan 1 文档

把本阶段的原因、结构、实现路径、常见问题和验收方式写清楚。

### 第八步：完成验收

确认项目能构建、能运行、能测试、文档也存在且内容一致。

## 十、代码层面后续会落到哪些位置

虽然 Plan 1 还不写完整业务逻辑，但要先知道后续代码会落在哪些位置：

1. `cmd/go-blockchain/main.go`：程序入口
2. `internal/cli/`：命令解析与命令组织
3. `internal/config/`：默认配置、配置加载
4. `internal/blockchain/`：后续区块、区块链、交易主线
5. `internal/wallet/`：后续钱包与地址
6. `internal/network/`：后续节点通信与同步
7. `internal/gui/`：后续 GUI 对接边界

这一步的意义是让目录从第一天开始就和未来目标一致。

## 十一、常见错误

### 错误 1：初始化阶段就写大量业务代码

这样会导致初始化阶段目标失控，也会让文档和代码都变得混乱。

### 错误 2：过早创建很多抽象目录

比如先建 `core`、`utils`、`common`，看起来整洁，后面其实很容易失控。

### 错误 3：没有统一 CLI 入口

如果后续每个命令都直接写在 `main.go`，项目很快会变得不可维护。

### 错误 4：测试全部堆到 `tests/`

Go 的单元测试更适合放在包目录下，用 `_test.go` 管理。

### 错误 5：文档只写标题，不解释原因

这种文档在答辩时帮助很小，因为老师真正关心的是你为什么这样设计。

## 十二、这一阶段怎么验收

Plan 1 完成后，至少要检查：

1. `go build ./...` 是否成功
2. `go run ./cmd/go-blockchain --help` 是否成功
3. `go run ./cmd/go-blockchain version` 是否成功
4. `go run ./cmd/go-blockchain doctor` 是否成功
5. `go test ./...` 是否成功
6. `docs/plan/README.md` 是否存在
7. `docs/plan/plan1.md` 是否存在
8. 目录结构是否和计划一致

## 十三、答辩时这一阶段可以怎么讲

如果老师问“为什么你一开始不直接做区块链功能”，你可以这样讲：

1. 这是一个逐步演进的大作业，不是一次性写完的小脚本
2. 后续会涉及区块、交易、UTXO、共识、网络、GUI 等多个模块
3. 如果没有统一的初始化骨架，后面会频繁重构，影响稳定性
4. 所以先建立可运行、可扩展、可讲解的项目骨架，是为了保证后续每个阶段都能自然接入

如果老师问“为什么文档从一开始就写”，你可以这样讲：

1. 文档不是事后总结，而是开发过程的一部分
2. 每个阶段同步写文档，能保证设计原因、理论依据和实现路径都不会丢
3. 最后答辩时，也能直接用这些文档支撑讲解

## 十四、本阶段完成后的下一步

Plan 1 完成后，下一步就可以进入基础区块链阶段。

到那时，项目已经具备：

1. 稳定目录结构
2. 统一入口
3. 配置基础
4. 文档体系
5. 基础测试规则

这样后面每推进一个功能阶段，都不会从混乱状态开始。
