# Plan 29：GUI 多签花费向导

## 一、这一阶段要解决什么问题

到 Plan 28 为止，GUI 已经可以：

1. 构造 P2PKH 交易
2. 构造 P2PK 交易
3. 构造教学型多签输出交易

但多签在 GUI 中还停留在“只能创建输出”的阶段。  
如果老师继续问“那多签输出能不能直接在 GUI 里花费”，之前还需要切回 CLI。

Plan 29 的目标，就是把多签从 GUI 中的“可创建”推进到“可花费”。

## 二、为什么这一阶段很重要

因为这一步让 GUI 真正打通了多签完整链路：

1. 创建多签输出
2. 查看当前未花费多签输出
3. 选择一个具体多签输出
4. 填入签名者并发起多签花费
5. 进入 mempool 并出块

这样答辩时，GUI 就不再只是脚本模板选择器，而是可以直接演示更复杂脚本流转的工作台。

## 三、本阶段范围

### 要做的内容

1. 在后端增加未花费多签输出查询能力
2. 在 GUI 服务层增加多签输出列表接口
3. 在 GUI 服务层增加多签花费交易排队接口
4. 在交易页增加“多签花费”区域
5. 让前端可以选择具体未花费多签输出
6. 允许填写签名者 CSV、目标地址、金额、手续费
7. 同步文档

### 这一阶段不做的内容

1. 不做复杂多签可视化流程图
2. 不做任意顺序签名匹配优化
3. 不实现完整脚本编辑器
4. 不实现更复杂的多签策略组合

## 四、核心理论讲解

### 1. 为什么 GUI 里要展示“未花费多签输出”

多签花费和普通转账不同，它不是只填一个 To 地址就够了。  
你必须明确告诉系统：

1. 要花哪一个具体输出
2. 这个输出的阈值是多少
3. 哪些参与者可以签名

所以 GUI 必须把当前未花费多签输出列出来，而不是让用户盲填。

### 2. 为什么签名者顺序仍然重要

当前项目采用教学型多签实现。  
为了保证实现简单、演示稳定，签名者 CSV 仍按锁定脚本中的公钥顺序匹配。  
这虽然比真实比特币语义更简化，但在课程项目中更容易讲明白，也更容易稳定验证。

### 3. 为什么多签花费继续走后端构造

前端的职责仍然只是：

1. 选输出
2. 收参数
3. 调后端

真正构造多签花费交易的逻辑依旧在 Go 后端中完成。  
这样能保证：

1. GUI 和 CLI 共享一套真实能力
2. 前端不会偷偷实现一套独立脚本规则
3. 结构保持低耦合

## 六、实现步骤

### 第一步：后端提供未花费多签输出查询

让系统能从当前 UTXO 视图中筛出未花费的多签输出。  
这是 GUI 能正确选择花费目标的前提。

### 第二步：GUI 服务层封装多签花费接口

让前端只传：

1. 签名者 CSV
2. 源交易 ID
3. 源输出下标
4. 收款地址
5. 金额
6. 手续费

### 第三步：交易页增加多签花费表单

交易页现在不仅能创建多签输出，也能直接花费多签输出。  
这样 GUI 中的脚本演示链路更完整。

## 七、GUI 如何演示

建议演示顺序：

1. 创建多个钱包
2. 用 GUI 交易页创建一个教学型多签输出
3. 挖矿确认
4. 在“多签花费”区域选择刚才的未花费多签输出
5. 填入签名者 CSV 和目标地址
6. 提交到 mempool
7. 再次挖矿并查看结果

演示重点：

1. GUI 已能看到具体的未花费多签输出
2. 多签花费不是假按钮，而是真正生成交易进入 mempool
3. GUI 和底层脚本引擎是同一套真实逻辑

## 八、验收标准

1. GUI 可列出未花费多签输出
2. GUI 可发起多签花费交易
3. 多签花费可进入 mempool
4. 挖矿后可在链上看到结果
5. 构建通过
6. 文档同步更新

## 九、常见错误

### 错误 1：前端自己推测多签输出而不读取真实后端状态

这样容易和真实 UTXO 视图不一致。

### 错误 2：多签花费仍然需要手动切回 CLI

那 GUI 的脚本交易工作台就还没闭环。

### 错误 3：为了做 GUI 花费把多签逻辑搬进前端

正确做法仍然是前端收参数，后端构造真实交易。
