# 区块链模拟仿真系统实验报告

## 一、实验题目

基于 Go 的区块链模拟仿真系统设计与实现

## 二、实验目的

本实验的目的是通过分阶段实现一个区块链模拟仿真系统，理解并掌握区块链中的关键机制，包括：

1. 区块与区块链结构
2. 钱包与地址系统
3. UTXO 交易模型
4. 签名与验证
5. Merkle 树
6. 工作量证明
7. 交易池与奖励机制
8. 多节点网络模拟
9. 安全校验与攻击模拟
10. 图形化展示与性能实验
11. Script / OP 虚拟机

## 三、系统总体设计

### 3.1 总体思路

系统采用“由简到繁、逐步演化”的开发方式：

1. 先实现基础链结构
2. 再实现交易
3. 再引入钱包、UTXO、签名
4. 再引入区块完整性与共识机制
5. 最后加入网络模拟、GUI 和性能实验

这样设计的优点是：

1. 每一阶段边界清晰
2. 容易调试和验证
3. 更适合课程答辩和原理讲解

### 3.2 模块划分

系统主要分为以下模块：

1. 区块链核心模块
2. 钱包模块
3. 网络模块
4. GUI 模块
5. 性能实验模块

## 四、核心数据结构设计

### 4.1 Block

区块结构包含：

1. 时间戳
2. 交易列表
3. `MerkleRoot`
4. 前一区块哈希
5. 当前区块哈希
6. 区块高度
7. `Nonce`
8. `Difficulty`

### 4.2 Transaction

交易结构包含：

1. 交易 ID
2. 输入集合
3. 输出集合

输入引用历史输出，输出锁定到收款方，符合 UTXO 风格设计。  
当前新交易还会显式写入：

1. `scriptSig`
2. `scriptPubKey`
3. 交易版本

### 4.3 Wallet

钱包结构封装：

1. ECDSA 私钥
2. 公钥
3. 地址生成逻辑

## 五、核心功能实现

### 5.1 钱包与地址

系统使用 ECDSA 生成密钥对，并通过公钥哈希、版本字节和校验和生成地址，再使用 Base58 编码输出。

### 5.2 UTXO 模型

系统采用 UTXO 模型而不是账户余额模型。  
余额通过未花费输出累计得到，交易输入必须引用历史输出，发送时支持自动找零。

### 5.3 签名与验证

普通交易必须签名，验证时会检查：

1. 引用输出是否存在
2. 当前公钥是否对应被花费输出
3. ECDSA 签名是否有效

### 5.4 Merkle 树

系统对区块内交易哈希构建 Merkle 树，最终生成 `MerkleRoot` 并写入区块头，从而增强区块完整性校验能力。

### 5.5 Script / OP 虚拟机

系统实现了教学型最小 Script VM，支持标准 P2PKH 验证路径，当前至少支持：

1. `OP_DUP`
2. `OP_HASH160`
3. `OP_EQUALVERIFY`
4. `OP_CHECKSIG`

这样交易验证不再只是直接写死在代码里的“比较公钥哈希 + 校验签名”，而是升级成了脚本执行流程。

### 5.6 工作量证明

系统实现了简化版 PoW：

1. 为区块增加难度和随机数
2. 通过不断尝试 `Nonce` 找到满足目标值条件的哈希
3. 出块时完成挖矿，验块时快速校验

### 5.7 Mempool 与经济模型

系统实现了：

1. 交易先进入 Mempool
2. 矿工通过 `mine` 命令从池中打包交易
3. 区块第一笔是 coinbase 奖励交易
4. 手续费最终计入矿工奖励

### 5.8 网络模拟

系统实现本地 TCP 多节点模拟，包括：

1. 节点启动
2. 节点发现
3. 区块同步
4. 交易广播
5. 区块广播

当前还支持更长分叉切换：当收到一条更长侧链时，系统会切换到更长链并重建 UTXO。

在此基础上，系统还会尝试恢复旧主链掉下来的、在新主链下仍然合法的普通交易，使其重新回到 Mempool。

系统还会记录最近一次链重组状态，供 CLI 和 GUI 查询与展示。

在此基础上，系统还会保存最近链事件列表，便于回顾最近几次链变化。

网络层还增加了孤儿块缓冲能力，用于应对父块未先到达的乱序区块场景。

最近链事件列表当前不仅记录 reorg，还记录创世初始化、主链增长和侧链块存储等事件。
GUI 网络页还可展示节点最近网络事件，例如 peer 加入、孤儿块缓存与恢复。
节点在 tip 前进后还会向其他 peer 发出最新高度通告。
GUI 还可直接对指定节点执行初始化链、提交交易和挖矿操作，便于演示分布式节点工作流。
当前 GUI 和内置控制台还提供一键双节点网络演示，可快速复现实验链路。

Script VM 在当前阶段已支持两种真实模板：

1. P2PKH
2. P2PK
3. 教学型多重签名

同时，GUI 交易页也已支持直接选择不同脚本模板来发起交易。
GUI 现阶段还支持直接选择并花费未花费多签输出。
Dashboard 也已重构为更接近桌面客户端的多区域总控台。

### 5.9 安全校验

系统可以拒绝：

1. 双花交易
2. 非法签名交易
3. 非法引用输出
4. 非法区块导入

并提供 `simdouble` 命令用于双花攻击演示。

### 5.10 GUI

系统实现了基于 Wails + React 19 + TypeScript + MUI 的桌面 GUI，支持：

1. Dashboard
2. 钱包页面
3. 区块浏览
4. 交易池和挖矿页面
5. Script VM 相关脚本可视化展示
6. 节点网络管理
7. 内置 CLI 控制台

## 六、性能实验

### 6.1 实验目的

验证 UTXO 缓存相较全链扫描的查询性能优势。

### 6.2 实验方法

使用 `runperf` 命令，对以下两种方式进行对比：

1. `BalanceOfByScan`：全链扫描
2. `BalanceOf`：UTXO 缓存查询

### 6.3 实验结果

实验结果已写入：

1. `docs/perf/latest.json`
2. `docs/perf/latest.md`

其中记录了：

1. 查询次数
2. 地址数量
3. 缓存耗时
4. 扫描耗时
5. 加速倍数

### 6.4 实验结论

在当前测试规模下，UTXO 缓存查询明显优于全链扫描，更适合后续区块数增大后的场景。

## 七、系统特点总结

本系统的主要特点包括：

1. 分阶段结构清晰
2. 区块链主线完整
3. 钱包、UTXO、签名、PoW、Merkle、Mempool、网络模拟、GUI 全部打通
4. 已加入教学型 Script / OP 虚拟机
5. 可进行双花和非法区块演示
6. 有可复现性能实验数据

## 八、存在不足

当前系统仍存在一些简化点：

1. 当前只实现了最小 P2PKH Script VM，还不是完整 Bitcoin Script
2. 网络层仍是教学型本地模拟
3. 分叉与最长链处理仍较简化
4. 动态难度调整尚未实现

## 九、实验收获

通过本实验，我系统性地理解了区块链从数据结构到交易验证、从共识到网络传播、从安全校验到 GUI 展示的完整实现链路。

本实验不仅提高了工程实现能力，也增强了对区块链核心机制的整体认识。
